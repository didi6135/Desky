# Task 6.2 — First real skill: `reminders`

**Status:** ⏳ pending
**Estimated effort:** ~2 hr
**Depends on:** 3.4.5.1 ✓ (skill data substrate), 5.1 (persona — for timezone), 5.0a + 5.0b (memory MCP)
**Blocks:** 6.3 (marketplace can install it as the canonical example)

## Goal

The first useful skill ships with Claudify. Uses everything: data
dir, persona, gotchas, progressive disclosure. Acts as the
**canonical reference** for skill authors.

End-state behaviour:

```
operator → "remind me to call Dani Thursday at 3pm"
bot      → ✓ saved to reminders.db; will ping Thursday 09:00
...Thursday 14:55...
bot      → (proactively, via Telegram) "Reminder: call Dani at 3pm"
```

## Why

Reminders are concrete, immediately useful, and exercise every
substrate piece. After it ships, the operator has a daily reason
to use the bot beyond message triage.

## Scope

### Files to create

- `skills/reminders/SKILL.md` — written per `docs/skills.md` template (≤ 500 lines, has Gotchas section, uses `${CLAUDIFY_SKILL_DATA}`)
- `skills/reminders/scripts/add.sh`
- `skills/reminders/scripts/list.sh`
- `skills/reminders/scripts/mark-done.sh`
- `~/.claudify-<n>/data/reminders/reminders.db` — schema below (created on first invocation via `memory_dir reminders` + sqlite3)
- systemd timer + service unit pair (per-instance) for the every-minute due-check loop

### Files to modify

- `install.sh` — register reminders skill via `manifest_set_skill <name> reminders '"reminders.db"' '["persona.db"]'`
- `CHANGELOG.md` — `### Added` entry

## Schema

```sql
CREATE TABLE reminders (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  text       TEXT NOT NULL,
  due_at     TEXT NOT NULL,        -- ISO 8601, operator's timezone
  created_at TEXT NOT NULL,
  notified   BOOLEAN DEFAULT 0
);
CREATE INDEX idx_reminders_due ON reminders(due_at) WHERE notified = 0;
```

## Steps

1. Write the SKILL.md per template (When-to-use / Procedure / Gotchas / Files-in-this-skill).
2. `scripts/add.sh "<text>" "<due_at>"` — calls `memory_dir reminders`, opens DB, INSERTs.
3. `scripts/list.sh` — SELECT all non-notified ordered by due_at.
4. `scripts/mark-done.sh <id>` — UPDATE notified=1.
5. systemd timer + service: every minute, scan for `due_at <= now AND notified=0`, send via Telegram, mark notified.
6. Reads operator timezone from persona.db via `memory.read /memories/persona.db` or `persona.get timezone` (MCP).
7. Register skill in manifest with write=reminders.db, read=persona.db.
8. CHANGELOG.

## Test plan

- **Local (bats):** add / list / mark-done — pure sqlite operations
- **Station11:** install reminders skill; create reminder for "60 seconds from now"; verify it fires within 90s; verify Telegram message arrives

## Out of scope

- Natural-language parsing of "Thursday at 3pm" — relies on the engine to interpret operator messages and pick correct ISO timestamps. Done by the LLM, not by add.sh.
- Recurring reminders ("every Friday") — bonus if it falls out, not required
- Multi-channel reminders (email, SMS) — Telegram only for now
- Cross-instance reminders (each instance has its own DB)

## Notes / risks

- **Timezone is the easy-to-mess-up part:** persona.db's `timezone` value (e.g. `Asia/Jerusalem`) drives all date math. The skill must respect it; tests must cover DST transitions.
- **Memory tie-in:** uses Tier-2 SQLite directly via the skill's data dir (per 3.4.5.1's `${CLAUDIFY_SKILL_DATA}`); reads persona.timezone via the MCP.
- **Systemd timer per-instance:** name it `claudify-reminders-<instance>.timer` to avoid collision with other instances on the same host. Lives in `~/.config/systemd/user/`.
- **Canonical reference role:** other skills will copy this structure. Keep it clean — no shortcuts that other skills shouldn't take.
