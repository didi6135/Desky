# Task 6.3 — Skill marketplace discovery + install UX

**Status:** ⏳ pending
**Estimated effort:** ~1.5 hr
**Depends on:** 3.4.5.1 ✓ (data dir + manifest schema), 3.4.6 (personal cmd), 6.2 (reminders — the canonical skill we install)
**Blocks:** 6.4 (template makes new skills installable via this flow)

## Goal

```
$ client-a skill install reminders
↓ resolves: github.com/claudify-skills/reminders
↓ validates SKILL.md present + structure OK
↓ lays under ~/.claudify-client-a/skills/reminders/
↓ runs config.json prompt if needed
↓ registers in claudify.json under skills[]
✓ installed: reminders @0.1.0
```

## Why

Pre-6.3, every skill ships in the Claudify repo itself
(`/skills/`). Post-6.3, third-party skills become installable —
the operator picks which skills they want. Marks the moment we
need a curated `claudify-skills` GitHub org.

## Scope

### Files to create

- `bin/claudify-skill` (or extend the personal cmd wrapper from 3.4.6 with a `skill` subcommand)
- `lib/skill-install.sh` — resolve / validate / lay-out / register

### Files to modify

- The personal cmd wrapper (3.4.6) — register `skill` subcommand with `install`, `list`, `remove` actions
- `CHANGELOG.md` — `### Added` entry

## Steps

1. Implement `<instance> skill install <id>`:
   - Resolve to `https://github.com/claudify-skills/<id>` by default
   - `git clone --depth 1` into a tmp dir
   - Validate: `SKILL.md` present, frontmatter parseable, no `scripts/` writing outside `data/`, declared memory in manifest format
   - Move into `~/.claudify-<n>/skills/<id>/`
   - If `config.json` present, prompt for required values; write to `data/<id>/config.json`
   - `manifest_set_skill <name> <id> [writes-json] [reads-json]` from the SKILL.md frontmatter
   - `memory_dir <id>` to ensure the data dir exists at mode 700
2. Implement `<instance> skill list` — read manifest, print skills.
3. Implement `<instance> skill remove <id>` — manifest_unregister_skill + rm -rf the skill dir (NOT the data dir — operator data persists)
4. Tests + bats.
5. CHANGELOG.

## Acceptance criteria

- [ ] `client-a skill install reminders` works end-to-end on a fresh Station11
- [ ] Bad skill (missing SKILL.md / writes outside data dir) refuses to install with a clear error
- [ ] Manifest registers the install with version (parsed from SKILL.md frontmatter)
- [ ] `client-a skill list` shows the registered skill
- [ ] `client-a skill remove reminders` clears the manifest entry + the skill dir (preserves `data/reminders/`)

## Test plan

- **Local (bats):** install / list / remove against a tmp fake-repo
- **Station11:** install the real `reminders` skill from `claudify-skills/reminders`; verify it works end-to-end

## Out of scope

- Skill version pinning / upgrade flow — Phase 7
- Skill signing / supply-chain verification — Phase 7+ (security & observability)
- Mirror to a non-GitHub marketplace — Phase 7+
- Auto-update of installed skills — Phase 7+

## Notes / risks

- **Validation is security-critical.** A malicious SKILL.md can still cause harm even with the assert helpers. Conservative validation: refuse anything outside the documented surface (allowed file types, no top-level scripts outside `scripts/`, no symlinks).
- **The curated GitHub org `claudify-skills`** needs to exist before 6.3 ships. Operator-runnable today via raw URL; org makes discoverability cleaner.
- **`remove` preserves data:** the operator's reminders survive `skill remove reminders` + `skill install reminders`. Only the skill code dir is replaced. Data lives at `data/<id>/` and only `uninstall.sh` wipes it.
