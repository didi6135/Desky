# Task 6.1 — DM pairing flow

**Status:** ⏳ pending
**Estimated effort:** ~1 hr
**Depends on:** 3.4.5 ✓ (multi-instance), 3.4.5.1 ✓ (data dir)
**Blocks:** none directly; quality-of-life win

## Goal

Adding a new allowed user becomes:

1. New user sends any message to the bot
2. Bot replies with a 4-digit pairing code, doesn't process the message
3. Operator (already in allowlist) sends `/approve 1234`
4. Bot adds the new user's ID to `access.json`, replies "added"

Replaces today's flow (operator manually edits `access.json` and restarts the service).

## Why

`access.json` hand-editing is the rough edge for adding people. After 6.1, onboarding a family member or business partner is a 30-second Telegram round-trip, not an SSH session.

## Scope

### Files to create

- `lib/access.sh` (or extend the existing access.json handling) with:
  - `pairing_create <user_id>` — generates 4-digit code, stores in `access.json.pending`, expires in 5 min
  - `pairing_approve <code>` — finds the pending entry, moves user_id to `allowFrom`, clears the pending
  - `pairing_revoke <user_id>` — removes from allowlist
- Built-in skill `dm-pairing/` (or hook) that triggers when an unknown sender messages

### Files to modify

- `access.json` schema — already has `"pending": {}` slot from 3.4.5; wire it up
- `CHANGELOG.md` — `### Added` entry

## Steps

1. Write `lib/access.sh` with the 3 pairing helpers.
2. Decide invocation shape: built-in skill OR engine hook on unknown sender. Skill is more portable; hook is faster. Lean skill.
3. Hook into the unknown-sender code path: when access.json doesn't list the sender, emit pairing code reply + write `pending[code] = {user_id, expires_at}`.
4. `/approve <code>` is a slash command (operator-side) that calls `pairing_approve`; replies with confirmation.
5. Tests for create + approve + revoke + expired-code rejection.
6. CHANGELOG.

## Acceptance criteria

- [ ] New user → pairing-code reply within 1 polling cycle
- [ ] `/approve <code>` from existing allowlisted user → addition lands in `access.json`, persists across restarts
- [ ] `/approve` from a non-allowlisted user is ignored (no privilege escalation)
- [ ] Failed pairing (wrong code, expired) replies with a helpful error

## Test plan

- **Local (bats):** create / approve / revoke / expired-code
- **Station11:** new Telegram user → pairing flow → can chat

## Out of scope

- Group chat policy (per access.json `dmPolicy`) — separate task
- Multiple pending pairings concurrent — supported by `pending: {code: ...}` map, but UX for collision is out of scope
- Auto-expire daemon — codes expire on next access check, no separate timer needed

## Notes / risks

- **Memory tie-in:** none directly; runs in parallel with Phase 5.
- **Code length:** 4 digits is 10,000 possibilities. With 5-min expiry + single-pending-per-user it's fine. Bump to 6 digits if brute-force ever becomes plausible.
- **Privilege check:** `pairing_approve` MUST verify the approving user is in `allowFrom`. The skill is invoked by the running bot, which already runs against the allowlist — but defense in depth.
