# Task 5.1.1 — `<instance> remember <fact>` operator command

**Status:** ⏳ pending
**Estimated effort:** ~30 min
**Depends on:** 5.1 (persona.sh exists), 3.4.6 (personal cmd wrapper — Phase 3)
**Blocks:** none

## Goal

Operator can save persona facts with a single command — deliberately
and explicitly. Anti-creep alternative to mem0's auto-extract.

```
$ client-a remember "preferred_meeting_time" "Friday morning"
✓ remembered: preferred_meeting_time = Friday morning
```

Bonus: natural-language form parses via the engine:

```
$ client-a remember "I prefer Friday meetings before noon"
✓ remembered: preferred_meeting_time = Friday morning
```

## Why

mem0-style auto-extract from every message creates a creeping memory
that the operator can't audit. The explicit `remember` command keeps
the operator in control: they decide what gets persisted. Single
command + Telegram-visible confirmation = clear feedback loop.

## Scope

### Files to create

- `lib/cmd-remember.sh` (or a built-in skill under `skills/remember/`) — the command implementation. Takes either `<key> <value>` form or natural-language string.

### Files to modify

- Personal cmd wrapper (3.4.6) — register `remember` as a subcommand
- `CHANGELOG.md` — `### Added` entry

## Steps

1. Decide implementation shape: `lib/cmd-remember.sh` (called by the personal cmd wrapper) OR a `skills/remember/` skill. Lean toward `lib/cmd-remember.sh` — built-in, doesn't need skill plumbing yet.
2. Implement the two-arg form first: `cmd_remember <key> <value>` calls `persona_set` from 5.1.
3. For natural-language form: detect when only one positional arg is given, hand it to the engine (via a one-shot `claude -p` call) with a small prompt asking it to return `key=value`. Parse and call `persona_set`.
4. Reply on Telegram with `✓ remembered: <key> = <value>` — this requires the wrapper to know how to send a reply, which it does via the running bot.
5. CHANGELOG: `### Added — <instance> remember <fact> command. Two forms: explicit '<key>' '<value>' OR natural-language single-string which the engine parses into a key/value.`

## Acceptance criteria

- [ ] Auto-render updates CLAUDE.md (via persona_set → engine_apply_persona)
- [ ] Replied to via Telegram with `✓ remembered: <key> = <value>`
- [ ] Bonus: natural-language input parses correctly via the engine

## Test plan

- **Local (bats):** explicit-form test (mock persona_set + check it was called with right args)
- **Station11:** send `client-a remember "x" "y"` via the personal cmd; verify DB row + CLAUDE.md update + Telegram reply

## Out of scope

- Editing existing facts via natural language ("change my timezone to Asia/Tokyo") — same plumbing applies, but the prompt design is bigger. Phase 7 if needed.
- Deleting facts via `forget <key>` — easy to add later.
- Bulk import from a YAML file — out of scope.

## Notes / risks

- **Natural-language parsing reliability:** sending one operator string to the engine with a "extract key/value" prompt can hallucinate. Mitigations: keep the prompt strict ("Output exactly `key=value` or 'NONE'"), fall back to two-arg form on parse failure, never silently invent facts.
- **Round-trip latency:** the natural-language form makes a `claude -p` call, which can take 2-5 seconds. The explicit two-arg form is instant. Document both.
- **Concurrency:** if the bot is processing a long message and the operator runs `remember`, both might try to write `persona.db`. SQLite WAL mode handles this — but verify with a small concurrency test.
