## The real body

```bash
engine_memory_setup() {
  step "Register memory MCP"
  local mcp_path="$CLAUDIFY_INSTANCE_DIR/bin/claudify-memory.js"
  run "mkdir -p $(dirname "$mcp_path")"
  run "cp $LIB_DIR/../src/mcp/memory/dist/index.js $mcp_path"
  run "claude mcp add claudify-memory bun run $mcp_path \
    --env CLAUDIFY_INSTANCE_DIR=$CLAUDIFY_INSTANCE_DIR"
  ok "memory MCP registered"
}
```

Idempotent: `claude mcp add` short-circuits if already registered.
The `bin/` dir under the per-instance root is where MCP binaries
live (not under `data/`, which is for runtime state).

## Notes / risks

- **Engine-agnostic call site:** `install.sh` calls `engine_memory_setup` with no args — the engine adapter decides how to register. A future Gemini adapter implements its own body; no changes to install.sh.
- **`bin/` location:** the per-instance `bin/` is the canonical place for MCP server binaries. It's NOT under `data/` (data is for runtime state, not executables). The install creates it lazily — the `mkdir -p` in `engine_memory_setup` handles it.
- **Bundle vs source:** `dist/mcp/memory/index.js` is the Bun-bundled output of `src/mcp/memory/index.ts` (built by 5.0a's `build.sh` changes). `engine_memory_setup` copies the bundled artifact, not source.
