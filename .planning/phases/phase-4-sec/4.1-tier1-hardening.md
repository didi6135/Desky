# Task 3.6.1 — Tier-1 systemd hardening (always-safe directives)

**Status:** ⏳ pending
**Estimated effort:** ~30 min
**Depends on:** 3.5 (paths must be stable)
**Blocks:** 3.6.2 (filesystem write-restriction builds on this)
**Research:** [`.planning/research/security.md`](../../research/security.md) Part C §1

## Goal

Take the systemd unit from a `systemd-analyze --user security`
exposure score of ~9.6 ("UNSAFE") to ~5.0 ("OK") by adding the set
of hardening directives that have **zero compatibility risk** for
our use case.

These directives drop powers the bot doesn't need. None of them
break Bun, Node, claude-code, the Telegram plugin, or our scripts.

## Why

The unit today has zero hardening directives. A prompt-injected
shell command, a buggy skill, or a leaked token has nothing
limiting blast radius. Tier-1 is the cheapest, safest win
available. See `.planning/research/security.md` Part B for the
threat model and Part C §1 for the directive list with rationale.

## The Tier-1 block

Adds to the `[Service]` section of the systemd unit:

```ini
# === Privilege restrictions (drop powers we don't need) ===
NoNewPrivileges=true
RestrictSUIDSGID=true
LockPersonality=true
RestrictRealtime=true

# === Filesystem isolation (read-only protect kernel + system state) ===
PrivateTmp=true
ProtectKernelTunables=true
ProtectKernelModules=true
ProtectKernelLogs=true
ProtectControlGroups=true
ProtectClock=true
ProtectHostname=true

# === Namespace restrictions (block container-style sandbox creation) ===
RestrictNamespaces=true

# === Resource limits (DoS / runaway-skill mitigation) ===
MemoryMax=1G
TasksMax=200
LimitNPROC=200
```

### What each directive does (one-liner)

| Directive | Plain meaning |
|---|---|
| `NoNewPrivileges` | Bot can never gain power. setuid binaries can't escalate. |
| `RestrictSUIDSGID` | Bot can't create files marked "run as someone else." |
| `LockPersonality` | Bot can't pretend to be a different OS. |
| `RestrictRealtime` | Bot can't grab CPU priority and freeze the server. |
| `PrivateTmp` | Bot's `/tmp/` is private; no cross-process snooping. |
| `ProtectKernelTunables` | `/proc/sys` read-only to bot. |
| `ProtectKernelModules` | Bot can't load kernel modules. |
| `ProtectKernelLogs` | Bot can't read kernel ring buffer (dmesg). |
| `ProtectControlGroups` | Bot can't reach into Linux process-grouping. |
| `ProtectClock` | Bot can't change the system clock. |
| `ProtectHostname` | Bot can't rename the server. |
| `RestrictNamespaces` | Bot can't create container sandboxes. |
| `MemoryMax=1G` | Cap memory at 1 GB. |
| `TasksMax=200` | Cap thread count. |
| `LimitNPROC=200` | Cap child process count. |

Resource limits (1G/200/200) are sized for a single-user PA with
maybe a handful of skills. If you ever hit them in real use,
revisit — but they're 10× normal load.

## Out of scope

- Filesystem write-restriction (3.6.2 — `ProtectHome=read-only`
  and `ReadWritePaths=`). That has compatibility risk and needs
  its own commit.
- Address families + syscall filter (3.6.3). Same — separate commit.
- Per-skill resource limits (Phase 5).
- `MemoryDenyWriteExecute=true` — explicitly **excluded**. Bun + V8
  use JIT; this would crash the bot at startup. Documented in the
  unit comment as "do not add."
- `ProtectSystem=strict` — possibly blocks `/usr` symlinks for the
  npm-global claude binary. Tier-1 covers most of the benefit
  without the risk.

## Notes / risks

- **The heredoc grows.** Make sure the heredoc terminator (`SVC`)
  isn't accidentally matched by any directive value.
- **`MemoryMax=1G` is enforced even on first run.** If the install
  ever needs more than 1 GB resident (npm install of a giant
  package), it could OOM. Realistic: claude + bun + a few MCPs
  use far less than 100 MB resident. 1 GB is generous.
- **`TasksMax`/`LimitNPROC` of 200** is also generous. claude
  spawns ~5-10 child processes (telegram plugin MCP, future
  claudify-memory MCP). 200 is 20× ceiling.
- **Station11 is the test.** Local syntax checks don't catch
  systemd-analyze regressions. Always round-trip on real systemd.
