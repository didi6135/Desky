## Goal

After this task, the bot runs in a **private mount namespace** where
only its own instance directory is visible. Other instances'
folders, `~/.ssh`, `~/.bashrc`, and the rest of `$HOME` are not
"read-protected" — they **don't exist** in the bot's filesystem
view at all.

This is the **single biggest security win** in the Phase 4
umbrella, and it's the actually-secure foundation for multi-client
deployments.

> **Note on the original plan:** This task was originally specified
> as `ProtectHome=read-only + ReadWritePaths=...`. That plan
> restricted writes only — reads of other instances' data still
> succeeded because all instances run as the same Linux user.
> [ADR 0006](../../decisions/0006-multi-client-isolation.md) supersedes that
> approach with mount-namespace isolation. See ADR 0006 for the
> full rationale; this spec captures the implementation.

## Why

Today, a compromised conversation or buggy skill has full write
access to the operator's entire home directory. With `bypassPermissions`
in effect (no per-tool prompts), there is nothing between the LLM's
decision to write a file and the file being written. Filesystem
write-restriction at the systemd layer is the right place to bound
this.

See `.planning/research/security.md` Part C §2 for rationale.

## The block

Adds to the `[Service]` section, after the Tier-1 directives from 4.1:

```ini
# === Mount-namespace isolation ===
# $HOME starts as an empty tmpfs in this unit's mount namespace;
# only the explicitly bound paths are visible. Other instances'
# folders (~/.claudify-<other-name>/), ~/.ssh, ~/.bashrc, and
# everything else under $HOME literally don't exist in the bot's
# filesystem view. Per ADR 0006.
ProtectHome=tmpfs
BindPaths=%h/.claudify-<INSTANCE_NAME>
BindReadOnlyPaths=%h/.npm-global %h/.bun

# Per-instance Claude state lives inside the per-instance dir,
# not in user-wide ~/.claude. CLAUDE_CONFIG_DIR redirects everything.
Environment=CLAUDE_CONFIG_DIR=%h/.claudify-<INSTANCE_NAME>/claude
```

`<INSTANCE_NAME>` is interpolated by `write_service` (e.g.
`%h/.claudify-client-a` for the client-a instance).

### What's bound and why

| Path | Mode | Why |
|---|---|---|
| `%h/.claudify-<name>` | read-write | This instance's everything: state, manifest, OAuth, workspace, channels, MCPs, skills, hooks, data, claude config |
| `%h/.npm-global` | read-only | Where the `claude` binary lives. Bot needs to *run* claude, not modify the binary. |
| `%h/.bun` | read-only | Bun runtime. Same reason. |

### What's invisible to the bot

Inside the bot's mount namespace, `$HOME` looks like:

```
$HOME/                              ← empty tmpfs
├── .claudify-<this-instance>/      ← read-write
├── .npm-global/                    ← read-only
└── .bun/                           ← read-only
```

Everything else is gone:

- `~/.claudify-<other-instance>/` — other clients' data
- `~/.ssh/` — SSH keys, known_hosts
- `~/.bashrc`, `~/.profile`, `~/.zshrc`
- `~/.gnupg/`, `~/.aws/`, `~/.config/`
- Operator's personal files
- **Anything else under `$HOME`**

A prompt-injected `find ~/ -name '*.env' -exec cat {} \;` returns
nothing useful — those paths don't exist in the bot's view.

### Why `CLAUDE_CONFIG_DIR`?

Claude Code reads `~/.claude/` (settings, plugins cache) and
`~/.claude.json` (project trust state) by default. Both are
user-wide. For multi-client isolation, every instance needs its
own copy. Anthropic supports this via `CLAUDE_CONFIG_DIR`;
pointing it at `%h/.claudify-<name>/claude/` puts that state inside
the instance dir (which is bound) rather than in `~/.claude/`
(which is NOT bound, so wouldn't be visible anyway).

`engine_seed_state` in `lib/engines/claude-code.sh` is updated by
3.4.5 to write into `${CLAUDIFY_INSTANCE_DIR}/claude/` instead of
`~/.claude/`. This task confirms the systemd `Environment=` line is
correctly set.

### What about `~/.bashrc`?

Today's `_npm_prefix_setup` appends a `PATH` export line to
`~/.bashrc`:

```bash
echo "export PATH=\"$NPM_PREFIX/bin:\$PATH\"" >> "$rc"
```

**That's at install time, not at runtime.** The bot itself never
writes to `.bashrc` — only `install.sh` does, and `install.sh` runs
as the operator's interactive shell, NOT under the systemd unit.
So `ProtectHome=tmpfs` doesn't affect this code path.

## Decisions

### `ProtectHome=tmpfs` vs `=read-only` vs `=true`

| Option | What it does | Choice |
|---|---|---|
| `ProtectHome=true` | Hides $HOME entirely (empty bind, no readback) — same effect as tmpfs but no writable workspace for tmp files | ❌ |
| `ProtectHome=read-only` | Bot can read all of $HOME; can write only to `ReadWritePaths` | ❌ Read leak across instances; superseded by ADR 0006 |
| **`ProtectHome=tmpfs`** | $HOME replaced with empty tmpfs; only `BindPaths`/`BindReadOnlyPaths` are visible | ✅ Per ADR 0006 |

### Why not `ProtectSystem=strict`?

`ProtectSystem=strict` makes `/usr`, `/boot`, `/etc`, and other
system dirs read-only. Tempting, BUT: the npm-global symlinks for
`claude` go through `/usr/local/bin/claude` on some setups. If
`/usr` is unwritable and we ever try a global claude reinstall via
the bot, it'd fail. Tier-1's `ProtectKernelTunables` +
`ProtectControlGroups` + `ProtectClock` + `ProtectHostname` cover
most of `ProtectSystem`'s value without the foot-gun.

We could revisit `ProtectSystem=full` (which excludes `/etc`) in
Phase 7 if a hardening guide identifies it as low-risk for our
shape.

### Why `BindPaths` not `ReadWritePaths`?

`ReadWritePaths` only works as an exception to a less-restrictive
`ProtectHome=` mode. With `ProtectHome=tmpfs`, the listed paths
must be explicitly *bound* in (not just made writable in an
otherwise-readable home). That's `BindPaths` (read-write) and
`BindReadOnlyPaths` (read-only).

## Reduced scope — what 4.2 actually does now (2026-05-10)

### Goal

Strip the broken namespace directives from `lib/service.sh::write_service` so the systemd unit text Claudify writes doesn't *claim* isolation it can't deliver on Ubuntu 24.04. ~5 min of work.

### Why

The directives parse without error on systemd 255 and `systemctl show` reports them as set, but `/proc/<pid>/ns/mnt` shows no namespace was created (AppArmor blocks unprivileged userns mounts on Ubuntu 24.04). Operators reading the unit file would believe the bot is sandboxed; it isn't. Honesty matters.

### What to remove

From `lib/service.sh::write_service`'s heredoc:

```ini
# REMOVE these lines:
ProtectHome=tmpfs
BindPaths=%h/.claudify-${INSTANCE_NAME}
BindReadOnlyPaths=%h/.npm-global %h/.bun
```

### What to keep

Everything else in the unit:
- `EnvironmentFile=…/credentials.env`
- `Environment=CLAUDIFY_INSTANCE_NAME=…`
- `Environment=CLAUDIFY_INSTANCE_DIR=…`
- `Environment=CLAUDE_CONFIG_DIR=…/claude`
- `Environment=TELEGRAM_STATE_DIR=…`
- `WorkingDirectory=…`
- The Tier-1 hardening directives from 4.1 (when they land — they actually work on Ubuntu 24.04)

### What real isolation looks like

For anyone who needs cross-instance / multi-tenant isolation: use the **container delivery shape** from [3.4.9](3.4.9-containerize.md). Containers wrap the bot in a kernel-enforced namespace + cgroups. Same Claudify code, different delivery wrapper.
