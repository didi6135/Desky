Honest documentation up front beats surprise later. Operators
should know:
- The bot has the same FS + network access *they* have
- `bypassPermissions` means anything that gets through the
  allowlist + the model can take real actions
- systemd hardening is what bounds those actions

## The new README section

Insert in README.md, after the "Update" section, before "How it works":

```markdown
## Security

Claudify runs as **your Linux user** on your server. The bot has
exactly the access *you* have — it can read your files, hit the
network, run programs. It cannot become root, mount filesystems,
load kernel modules, or escape its sandbox.

**What's protected (Phase 3.6 hardening):**
- The bot can only **write** to four paths:
  `~/.claudify`, `~/.claude`, `~/.npm-global`, `~/.bun`. Everything
  else under your home is read-only to it. A misbehaving message or
  buggy skill cannot delete `~/.ssh`, edit `~/.bashrc`, or reach
  into anyone else's home.
- The bot cannot use raw network sockets, kernel-modification
  syscalls, or other unusual Linux features. Only HTTPS to Telegram
  + Anthropic, plus stdio between its own subprocesses.
- The bot's process group is capped: 1 GB memory, 200 threads, 200
  child processes. A runaway skill or fork-bomb gets killed by
  Linux instead of taking the server down.
- Secrets (`credentials.env`, bot token, allowlist) live in
  chmod-600 files, owned only by you.
- Only people on your allowlist can reach the bot through Telegram.
  Anyone else's messages are dropped at the channel layer.

**What's NOT protected:**
- A compromise of your Telegram account is a compromise of the
  bot. Anyone who can post as you, with `--permission-mode
  bypassPermissions` in effect, can ask the bot to take actions
  inside the sandbox. Treat your Telegram account as a credential.
- A compromise of your server (stolen SSH key, kernel CVE) is a
  compromise of everything you own. The bot's hardening doesn't
  defend against that — that's OS hardening.
- Side-channel attacks, supply-chain attacks against Anthropic /
  Telegram themselves, and physical access are out of scope.

**What you should do:**
- Run Claudify on a server **you control** with an allowlist **you
  trust**.
- Periodically run `bash <(curl -fsSL …/doctor.sh)` — the
  Security section catches hardening drift.
- If you ever paste a token into Telegram by mistake, *revoke it
  immediately* (BotFather for the Telegram bot token; `claude
  /logout` on your local machine for the OAuth token).
- For full details: `docs/architecture.md` §11 has the threat
  model; `.planning/research/security.md` has the audit and the
  hardening plan.
```

# ─── Contract: engine_run_args ────────────────────────────────────────────
# Echo the full ExecStart command for the systemd unit. Each engine
# decides whether it needs script(1) wrapping (Claude Code's TUI
# does — without a real PTY the channel plugin never spawns).

# Today's wiring: Telegram via plugin:telegram@claude-plugins-official.
# 3.4.4 (manifest) will pass channel/plugin set as args.
#
# === Security note: --permission-mode bypassPermissions ===
#
# This flag tells claude-code NOT to prompt for approval before each
# tool call. It's REQUIRED for an unattended Telegram bot — there's
# no operator at the terminal to click "approve" on every action.
#
# What this means:
#   - Anyone who can make claude take an action (i.e., anyone whose
#     message gets past the allowlist + the model's own judgement)
#     can in principle make claude run any tool the model has access
#     to: bash, edit files, web fetch, etc.
#   - The defense is upstream of claude:
#       1. Telegram allowlist (lib/configs.sh access.json)
#       2. systemd unit hardening (lib/service.sh — 3.6.x):
#          - ProtectHome=read-only + ReadWritePaths bound writes
#          - RestrictAddressFamilies + SystemCallFilter bound syscalls
#          - MemoryMax/TasksMax/LimitNPROC bound resource use
#       3. Curated skill set — operator chose what's installed
#       4. Persona / system prompt asks the model to confirm
#          destructive actions
#
# See .planning/research/security.md for the full threat model and
# what's deliberately NOT defended against.

}
