## Goal

Lock the bot to the network types and Linux system calls it actually
uses. Anything else ‚Äî raw packet sockets, esoteric kernel calls,
weird IPC families ‚Äî is denied at the kernel boundary.

After this task, `systemd-analyze --user security` lands at ~2.0
("OK") ‚Äî comparable to hardened sshd / nginx.

## Why

Tier-1 dropped powers the bot doesn't need. Tier-2 restricted where
the bot can write. Tier-3 restricts *how* the bot can talk to the
kernel. Together they shrink the bot's attack surface to
roughly: HTTPS to two endpoints + filesystem ops on four paths +
~150 standard syscalls.

This tier has higher compatibility risk than the previous two ‚Äî
Bun + Node + claude-code link against system libraries that may use
unusual syscalls. Hence: separate commit, full Station11 round-trip
including reply test.

## The block

Adds to `[Service]`, after Tier-2 (3.6.2):

```ini
# === Address-family restriction ===
# AF_UNIX: how claude talks to its MCP subprocesses (telegram plugin,
#          future claudify-memory) over stdio.
# AF_INET / AF_INET6: HTTPS to api.telegram.org and api.anthropic.com.
# Everything else (AF_NETLINK, AF_PACKET, AF_BLUETOOTH, ‚Ä¶) is denied.
RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6

# === System-call filter ===
# @system-service is systemd's pre-curated allowlist of ~150 syscalls
# normal daemons need (file ops, network, processes, signals).
# Blocks ~250 unusual ones: kexec, bpf, swapon, userfaultfd, mount,
# umount, reboot, etc.
SystemCallFilter=@system-service
SystemCallErrorNumber=EPERM
```

### What's blocked

`RestrictAddressFamilies` blocks (among others):
- `AF_PACKET` ‚Äî raw Ethernet (sniffing)
- `AF_NETLINK` ‚Äî kernel-routing socket
- `AF_BLUETOOTH`, `AF_VSOCK`, `AF_RDS`, etc.

`SystemCallFilter=@system-service` blocks:
- `kexec_load` (boot another kernel)
- `bpf` (load BPF programs)
- `swapon` / `swapoff`
- `mount` / `umount` / `pivot_root`
- `reboot`
- `userfaultfd`
- `lookup_dcookie`
- `delete_module` / `init_module`
- ~240 others most apps never use

`SystemCallErrorNumber=EPERM` makes blocked calls return "permission
denied" instead of killing the process ‚Äî apps that probe for
features (e.g. `bun` checking for `io_uring` support) handle EPERM
gracefully whereas a SIGSYS would kill them.

## Notes / risks

- **This tier has real compatibility risk.** Test path is the most
  important part. If Bun's V8 needs a syscall that's not in
  `@system-service`, the bot will fail to start or crash on first
  tool call. The journalctl scan in step 4 is what catches this.
- **`@system-service` is conservative.** systemd ships several other
  groups: `@network-io`, `@ipc`, `@process`. We could be more
  permissive (`@system-service @debug`) if we hit issues ‚Äî but
  start with the tightest acceptable set.
- **`AF_UNIX` is non-negotiable.** MCP stdio between claude and its
  subprocesses (telegram plugin, future claudify-memory) uses unix
  sockets. Without this, plugins fail to connect.
- **`AF_INET6` matters for Telegram.** Some Telegram endpoints are
  IPv6-only depending on routing. Don't drop it.
- **No `IPAddressAllow=` for now.** Telegram + Anthropic use dynamic
  IPs (CDN, geo-routing). Hardcoding ranges would break the bot
  every few months. Phase 5+ may revisit with DNS-based filtering
  or a proxy.
- **Future engines.** A future Gemini adapter might need different
  syscalls (e.g. CUDA-related ioctls if it ever runs locally). The
  filter should ideally be engine-contributed via a future
  `engine_systemd_extra` function ‚Äî but that's a refactor for
  later. For now, the filter is set assuming Claude Code only.
