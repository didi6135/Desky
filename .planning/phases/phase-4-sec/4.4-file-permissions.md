# Task 3.6.4 — Tighten file permissions (`access.json` + install log)

**Status:** ⏳ pending
**Estimated effort:** ~15 min
**Depends on:** 3.5 (paths stable)
**Blocks:** —
**Research:** [`.planning/research/security.md`](../../research/security.md) Part C §4

## Goal

Two cheap defense-in-depth wins:

1. **`access.json` → chmod 600** (was 644)
2. **`/tmp/claudify-install-*.log` → chmod 600 + `umask 077` before
   creation**

Neither file contains real secrets, but both contain
operationally-sensitive information (allowlist user IDs, install
context). On a multi-user server they leak; on a single-user server
they don't matter — but it's a one-line fix.

## Why

`docs/architecture.md §11` says everything bot-owned should be
operator-readable only. Today these two files default to 644,
visible to any user on the box. Fix it.

## Scope

### Files to modify

- `lib/configs.sh::_write_access_json` — chmod 600 after write
- `lib/ui.sh::setup_logging` — `umask 077` + chmod 600 on the log
  file
- `CHANGELOG.md` — `### Security` entry

### Files to create

- none

## The changes

### 1. `_write_access_json` in `lib/configs.sh`

Today (the `cat > ... <<JSON` and `jq` paths):

```bash
cat > "$access" <<JSON
{
  "dmPolicy": "allowlist",
  ...
}
JSON
ok "allowlist written (user $TG_USER_ID)"
```

After:

```bash
cat > "$access" <<JSON
{
  "dmPolicy": "allowlist",
  ...
}
JSON
chmod 600 "$access"
ok "allowlist written (user $TG_USER_ID)"
```

And in the jq-merge path:

```bash
mv "$tmp" "$access"
chmod 600 "$access"
ok "added $TG_USER_ID to existing allowlist"
```

`mv` from `mktemp` may preserve a 600-friendly mode but explicit is
better.

### 2. `setup_logging` in `lib/ui.sh`

Today:

```bash
LOG_FILE="${LOG_FILE:-/tmp/claudify-install-$(date +%Y%m%d-%H%M%S).log}"

setup_logging() {
  exec > >(tee -a "$LOG_FILE") 2>&1
}
```

After:

```bash
LOG_FILE="${LOG_FILE:-/tmp/claudify-install-$(date +%Y%m%d-%H%M%S).log}"

setup_logging() {
  # Log may contain operationally sensitive context (paths, install
  # decisions, redacted-but-still-detailed output). chmod 600 + umask
  # 077 before any tee writes so even a brief window where the file
  # exists at 644 is closed.
  umask 077
  : > "$LOG_FILE"
  chmod 600 "$LOG_FILE"
  exec > >(tee -a "$LOG_FILE") 2>&1
}
```

The `: > "$LOG_FILE"` creates the file empty under the tightened
umask; `chmod 600` is belt-and-suspenders.

## Steps

1. Edit `lib/configs.sh::_write_access_json` — add `chmod 600
   "$access"` after each successful write (both the create and
   merge paths).
2. Edit `lib/ui.sh::setup_logging` — add `umask 077`, pre-touch,
   `chmod 600` before the `exec >` line.

## Notes / risks

- **Setting `umask 077` in `setup_logging`** affects subsequent
  file creates in the install. We should verify nothing else in the
  install relies on default 022 umask. Looking through:
  - `_write_bot_env` — explicit `chmod 600` after, irrelevant
  - `_write_access_json` — was default, now explicit 600
  - `_seed_persona` — explicit `chmod 644` after, irrelevant
  - manifest writes — explicit, irrelevant
  - `engine_seed_state` jq writes — these go to ~/.claude.json + ~/.claude/settings.json; checking what mode they end up at... probably 600 from inherit, which is fine for our user but might surprise `claude` if it expects 644. **Worth a Station11 verification.**
- The `umask 077` in `_persist_oauth_token` already exists and is
  scoped via subshell; it's not affected.
- Setting umask in `setup_logging` (which runs once at install
  start) means *all subsequent files in the install run* default to
  600 unless explicit. Check whether any install-time file should
  be 644 (none come to mind, but verify).
