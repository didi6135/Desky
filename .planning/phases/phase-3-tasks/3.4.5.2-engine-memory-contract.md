## Why

The engine-agnostic rule (newly explicit in `CLAUDE.md`) requires
that any engine-specific surface live behind an `engine_*` function.
Memory has *two* engine-specific surface points (tool registration
+ persona injection), so two new contract functions.

Adding the contract stubs in Phase 3 means:
1. Phase 4.0a / 4.1 can implement *only the bodies* without
   redesigning the contract mid-flight
2. The engines/README.md documentation stabilises early
3. A future Gemini adapter has a clear stub-shaped reference

This is a plumbing-only task. The actual memory work is Phase 4.

### `engine_apply_persona <text>`

**Args:** the rendered persona text (a small markdown block).

**Behaviour:** make the rendered persona text part of every model
session. **Idempotent** — re-running with the same text leaves the
file unchanged. Re-running with new text replaces only the marked
block, preserving the operator's additions outside it.

**Implementation (Claude Code adapter):**
```bash
engine_apply_persona() {
  local rendered="$1"
  local target="$CLAUDIFY_INSTANCE_DIR/workspace/CLAUDE.md"
  mkdir -p "$(dirname "$target")"

  local marker_start='<!-- claudify:persona:start -->'
  local marker_end='<!-- claudify:persona:end -->'
  local block
  block="$(printf '%s\n%s\n%s\n' "$marker_start" "$rendered" "$marker_end")"

  if [[ -s "$target" ]] && grep -q "$marker_start" "$target"; then
    # Replace the existing block in place; awk preserves rest of file.
    awk -v start="$marker_start" -v end="$marker_end" -v new="$block" '
      $0 ~ start { print new; in_block=1; next }
      in_block && $0 ~ end { in_block=0; next }
      in_block { next }
      { print }
    ' "$target" > "$target.tmp" && mv "$target.tmp" "$target"
  else
    # First time: prepend the block above any existing content.
    if [[ -s "$target" ]]; then
      printf '%s\n\n%s\n' "$block" "$(cat "$target")" > "$target.tmp"
      mv "$target.tmp" "$target"
    else
      printf '%s\n' "$block" > "$target"
    fi
  fi
  chmod 644 "$target"
}
```

The marker pair is what makes this safe to call repeatedly without
clobbering operator edits.

## Notes / risks

- **Marker idempotency** is the load-bearing design choice. If the
  awk replace is buggy, repeated `engine_apply_persona` calls could
  duplicate or corrupt the block. Step 8 explicitly tests the
  re-run path.
- **First-time call on an existing CLAUDE.md** (operator already has
  `seed_persona` content there from Phase 3.3): the block prepends
  above the existing content. The persona auto-render becomes the
  *first* thing the model sees. Operator's hand-edits stay intact
  below.
- The `engine_memory_setup` stub returning 0 means `install.sh` can
  call it unconditionally. Phase 4.0b will replace the body without
  touching the call site.
