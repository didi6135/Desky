## What landed

Multi-instance bash install. Each install lives at
`~/.claudify-<name>/`, runs as `claudify-<name>.service`, registered
in `~/.claudify-registry.json`. Per-instance Claude state via
`CLAUDE_CONFIG_DIR`. Bot-token collision check across instances.
Tier-1 systemd hardening (the directives that actually work on
Ubuntu 24.04 user-mode).

**What did NOT land:** mount-namespace cross-instance isolation. That
turned out to be blocked by Ubuntu 24.04's
`kernel.apparmor_restrict_unprivileged_userns=1` and was honestly
dropped from this task. Real isolation is now [3.4.9 Containerize
Claudify](3.4.9-containerize.md)'s responsibility.

## Design choices

| Decision | Why |
|---|---|
| **Flat layout (`~/.claudify-<name>/`), not nested (`~/.claudify/instances/<name>/`)** | Per ADR 0006 Decision 1. Operator can `cd` directly to an instance. Clean `rm -rf`. Maps naturally to a future container's volume. |
| **Explicit `--name`, no auto-renaming** | Per ADR 0006 Decision 3. Operator types `--name client-a` once = clear identifier forever. Auto-numbering (`-2`, `-3`) defeats psychological isolation. |
| **Side-car registry at `~/.claudify-registry.json`** | Per ADR 0006. Lets `update.sh --all` and `doctor.sh` enumerate without a top-level container dir. |
| **Per-instance `CLAUDE_CONFIG_DIR`** | Anthropic's documented way to redirect Claude Code's state. Means each instance has its own `claude.json`, `settings.json`, plugin cache, project trust. No cross-instance leakage in Claude's own state. |
| **`CLAUDE_CONFIG_DIR` exported at the top of install.sh main()** | install-time `claude` invocations (plugin install, status checks) honor the per-instance dir from the start. If we set it only in the unit, the plugin would land in the wrong place. |
| **Bot-token collision check across instances** | Telegram allows only one polling client per token. Two instances on the same token silently fight. Explicit refusal at install time prevents the foot-gun. |
| **Dropped `WORKSPACE` prompt** | The instance name IS the workspace identifier. One fewer concept; simpler UX. |
| **Mount-namespace isolation dropped after Station11 experiment** | `ProtectHome=tmpfs + BindPaths` silently no-op on Ubuntu 24.04 user-mode systemd (AppArmor blocks unprivileged userns mount ops). Adding `PrivateUsers=yes + PrivateMounts=yes` didn't help — same AppArmor wall. Documented in ADR 0006 appendix. Real isolation moves to containers (3.4.9). |
| **Tier-1 hardening trimmed to capability-free directives** | `ProtectKernelModules`, `ProtectClock`, `ProtectHostname`, `ProtectKernelLogs` need CAP_SETPCAP to drop kernel capabilities — user-mode systemd lacks it. They fail with `status=218/CAPABILITIES`. Kept: `NoNewPrivileges`, `RestrictSUIDSGID`, `LockPersonality`, `RestrictRealtime`, `RestrictNamespaces` (all seccomp-based, no caps needed), plus `MemoryMax`, `TasksMax`, `LimitNPROC` (cgroup-based, no caps needed). |

## Station11 round-trip

Two-phase verification:

### Phase 1 — install + layout verification

```text
scrub everything
install client-a with --non-interactive
→ install completes cleanly
→ ~/.claudify-client-a/ exists with full subdirectory tree
  (workspace, channels, mcps, skills, hooks, data, claude)
→ ~/.claudify-registry.json contains client-a
→ ~/.config/systemd/user/claudify-client-a.service exists
→ systemctl --user is-active = "active"
```

### Phase 2 — hardening verification

```text
systemctl --user show claudify-client-a -p NoNewPrivileges -p MemoryMax …
→ NoNewPrivileges=yes
→ RestrictSUIDSGID=yes
→ LockPersonality=yes
→ RestrictRealtime=yes
→ RestrictNamespaces=yes
→ MemoryMax=1073741824 (1 GiB)
→ TasksMax=200
→ LimitNPROC=200

cgroup verification:
→ /sys/fs/cgroup<unit>/memory.max = 1073741824
→ /sys/fs/cgroup<unit>/pids.max   = 200
(kernel-enforced caps — verified)

bot Telegram check:
→ getUpdates returns 409 (long-poll conflict = bot actively polling)
→ live TCP connection to 149.154.166.110:443 (Telegram API)

doctor.sh --name client-a
→ 35 passed, 0 failures
```

### Failed experiments (recorded in ADR 0006 appendix)

- **`ProtectHome=tmpfs + BindPaths + BindReadOnlyPaths`**: parsed by systemd, reported by `systemctl show`, but `/proc/<pid>/ns/mnt` was identical to the operator's. No namespace created. Cause: AppArmor.
- **`PrivateUsers=yes` + `PrivateMounts=yes`**: user namespace created (different `ns/user` ID), mount namespace still not created. Same AppArmor wall.
- **`ProtectKernelModules`, `ProtectKernelLogs`, `ProtectClock`, `ProtectHostname`**: caused `status=218/CAPABILITIES` failures because user-mode systemd lacks CAP_SETPCAP. Dropped from the directive set.
