# Task 3.4.5 — Multi-instance layout + `--name` flag everywhere

**Status:** ✅ DONE 2026-05-11
**Estimated effort:** ~1.5 hr
**Depends on:** 3.4.3 (engine), 3.4.4 (manifest)
**Blocks:** 3.4.6 (personal command needs the instance name), 3.4.7 (migration)

## Goal

Move from the single-instance layout (`~/.claudify/...`) to the
multi-instance layout (`~/.claudify/instances/<name>/...`). Every
entrypoint accepts `--name <NAME>` (default: `default`). systemd
service per instance: `claudify-<name>.service`.

## Why

`docs/architecture.md §3b` describes the runtime layout. `§5a`
defines the registry. Multi-instance is a core invariant of the
architecture; until paths nest under `instances/<name>/`, the rest
is documentation without code.

## Scope

### Files to modify

- `lib/configs.sh` — paths become `~/.claudify/instances/$INSTANCE_NAME/...`
- `lib/oauth.sh` — same
- `lib/service.sh` — unit name becomes `claudify-$INSTANCE_NAME.service`; `WorkingDirectory` per-instance
- `lib/onboarding.sh` — `collect_inputs` adds an instance-name prompt (or uses env `INSTANCE_NAME`); validates per `§3d`
- `lib/validate.sh` — add `validate_instance_name` (regex + blocklist of common Unix commands)
- `lib/args.sh` — add `--name` flag to install
- `lib/manifest.sh` — already path-aware from 3.4.4; double-check it uses the new paths
- `install.sh`, `update.sh`, `uninstall.sh`, `doctor.sh` — all gain `--name` flag (default: `default`)
- `lib/preflight.sh` — `preflight_linger` is per-user not per-instance; no change. Other checks may need to be per-instance.
- `build.sh` — no change (already concatenates lib/)
- `lib/README.md` — update function table for any changed signatures

### Files to create

- `tests/bash/multi-instance.bats` — install two instances side-by-side; confirm independence

### Files to delete

- none

## Key design points (from architecture.md §3b)

- Default instance name: `default`
- Path pattern: `~/.claudify/instances/<name>/`
- Service name: `claudify-<name>.service`
- Working dir: `~/.claudify/instances/<name>/workspace`
- Manifest: `~/.claudify/instances/<name>/claudify.json`
- Registry update: `manifest_register_instance <name>`

## Steps

1. Add `INSTANCE_NAME` global (set by `parse_args`, default `default`)
2. Define `claudify_instance_dir()` helper in `lib/manifest.sh`: returns `$HOME/.claudify/instances/$INSTANCE_NAME`
3. Replace every hard-coded path in `lib/configs.sh`, `lib/oauth.sh`, `lib/service.sh`, `lib/onboarding.sh` (where it touches `~/.claudify/...`) with `$(claudify_instance_dir)/...`
4. Service unit name: `claudify-$INSTANCE_NAME.service` everywhere it's referenced (`write_service`, `start_service`, `final_summary` if it shows the unit name)
5. Add `validate_instance_name` to `lib/validate.sh`:
   - Regex: `^[a-z][a-z0-9_-]{1,30}$`
   - Blocklist (env BLOCKLIST or hard-coded array): `ls cd cp mv rm cat grep find git npm bun node claude claudify docker systemctl journalctl sudo bash sh zsh`
6. `lib/args.sh` — add `--name` parsing; validate via `validate_instance_name`
7. `install.sh`, `update.sh`, `uninstall.sh`, `doctor.sh` — accept and forward `--name` (uninstall: `--name` becomes *required* unless `--all` is passed)
8. `doctor.sh` without `--name` — reads registry, shows a one-line summary per instance
9. Add `tests/bash/multi-instance.bats`:
   - Install instance `alpha` with fake env vars + `--non-interactive`
   - Install instance `beta` with different fake env vars
   - Confirm both `~/.claudify/instances/alpha/` and `.../beta/` exist with separate state
   - Confirm two systemd units; both startable; neither stomps the other
   - (Skip cleanly on systems without bats)
10. `bash test.sh` — green
11. **Station11 round-trip:**
    - `bash ~/test/uninstall.sh --name default --yes`
    - `bash ~/test/autoinstall.sh` — installs as `default` by default
    - `bash ~/test/doctor.sh` (no flag) — reports `default: ✓`
    - `bash ~/test/doctor.sh --name default` — same, but explicit
    - Bonus: `bash ~/test/autoinstall.sh` with a different `--name` would create a second instance — for now just confirm the first one works
12. CHANGELOG: `### Added — multi-instance support: every install is named (default: 'default'), runs under its own systemd unit (claudify-<name>.service), and lives in ~/.claudify/instances/<name>/.`

## Acceptance criteria

- [ ] systemd unit naming: `claudify-<name>.service`

## Test plan

- **Local:** bats: install two named instances, assert directory + unit independence
- **Station11:** install + doctor on `default`; bot still replies
- **Station11 (advanced, optional now):** install second instance with `--name beta`, confirm both work simultaneously

## Out of scope

- Migration of an existing single-instance layout to multi-instance (3.4.7)
- Personal command wrappers (3.4.6)
- TypeScript backup/restore aware of instances (3.5)

## Notes / risks

- The `default` default name is important. Don't make `--name` required for install — that's a UX regression for single-instance users. Required only for uninstall (where ambiguity is dangerous).
- Two instances using the same Telegram bot token is a runtime conflict (Telegram API allows only one polling client per token). Validate at install: refuse if the bot token is already in another instance's `.env`.
- The `claudify-<name>.service` naming change is breaking for existing installs. 3.4.7 (migration) is what handles that. Until 3.4.7 lands, **the new install layout is incompatible with the old layout**. Don't deploy 3.4.5 alone to Station11; bundle 3.4.5 + 3.4.7 in the same release.

### 1. Create the FULL extension-type subdirectory tree per instance

Today's spec creates `instances/<name>/` and moves a couple of
existing dirs into it. Per `docs/architecture.md §4` and
`docs/skills.md §2`, every instance has 4 extension-type subdirs
plus `data/` and `workspace/`. **Create them all at install time,
even when empty.** Empty placeholders make the convention real,
make manifests honest, and unblock 3.4.5.1 + every Phase 4 task
without extra mkdir calls scattered later.

```
~/.claudify/instances/<name>/
├── workspace/        # engine WorkingDirectory (CLAUDE.md persona)
├── channels/         # ← MOVED from ~/.claudify/telegram (see #2)
│   └── telegram/     # the only channel today
├── mcps/             # MCPs install here (Phase 4)
├── skills/           # skills install here (Phase 4)
├── hooks/            # skill-bundled hooks (Phase 4)
├── data/             # per-skill data dirs (3.4.5.1 + Phase 4)
└── claudify.json     # per-instance manifest (was at root in 3.4.4)
```

`install.sh` `mkdir -p`s all six (channels/, mcps/, skills/, hooks/,
data/, workspace/) under `instances/<name>/` as part of the layout
step.

### 2. `channels/telegram/` replaces `telegram/`

Today: `~/.claudify/telegram/.env` and `~/.claudify/telegram/access.json`.
After 3.4.5: `~/.claudify/instances/<name>/channels/telegram/.env`
and `…/channels/telegram/access.json`.

This reflects **architecture.md §4a's intent**: channels are one of
the 4 extension types, and they nest under `channels/<name>/`.
3.4.5's current spec is silent on this — explicit add.

Affected:
- `lib/configs.sh` — `CLAUDIFY_TELEGRAM` resolves to
  `${CLAUDIFY_INSTANCE_DIR}/channels/telegram`
- Systemd unit's `TELEGRAM_STATE_DIR=` Environment line points at the
  new path
- `doctor.sh` checks the new path
- 3.4.7 migration moves the dir as part of layout migration

### 3. Flip the TODO(3.4.5) marker in `lib/manifest.sh`

3.4.4 left a one-line marker:

```bash
_instance_manifest_path() {
  local name="${1:-default}"
  # TODO(3.4.5): printf '%s/instances/%s/claudify.json' "$CLAUDIFY_ROOT" "$name"
  if [[ "$name" == "default" ]]; then
    printf '%s/claudify.json' "$CLAUDIFY_ROOT"
  else
    printf '%s/claudify-%s.json' "$CLAUDIFY_ROOT" "$name"
  fi
}
```

3.4.5 collapses both branches to the post-migration form:

```bash
_instance_manifest_path() {
  printf '%s/instances/%s/claudify.json' "$CLAUDIFY_ROOT" "${1:-default}"
}
```

One-line edit in `manifest.sh`. 3.4.7 (migration) handles converting
existing `~/.claudify/claudify.json` to the new path on upgrade.

### 4. Export `CLAUDIFY_INSTANCE_NAME` + `CLAUDIFY_INSTANCE_DIR` in the systemd unit

Skills, MCPs, and hooks all need to know which instance they're
running under. Two new `Environment=` lines in the systemd unit:

```systemd
Environment=CLAUDIFY_INSTANCE_NAME=<name>
Environment=CLAUDIFY_INSTANCE_DIR=%h/.claudify/instances/<name>
```

These propagate to the engine subprocess and from there to whatever
skill / MCP it invokes. Matches Anthropic's `${CLAUDE_PLUGIN_DATA}`
convention from the skill engineering blog post (we extend it with
`${CLAUDIFY_SKILL_DATA}` per-skill in 3.4.5.1).

`engine_run_args` in `lib/engines/claude-code.sh` doesn't need to
change — these go in the unit's `[Service]` section, not the
ExecStart command.

### 5. Drop the `WORKSPACE` prompt; default to `$INSTANCE_NAME`

Today onboarding asks for a `WORKSPACE` folder name (default
`claude-bot`). With multi-instance, the *instance name* is the
identifier — `WORKSPACE` becomes redundant noise. Two options:

- **(a)** Drop the WORKSPACE prompt entirely; use `$INSTANCE_NAME`
  wherever `$WORKSPACE` was used. **Simpler — recommended.**
- **(b)** Keep WORKSPACE as an optional "display name" override,
  defaulting to `$INSTANCE_NAME`. More complexity for marginal value.

Going with (a). One fewer prompt, one less concept the operator
juggles. Matches the simplicity goal.

### 6. `doctor.sh` iterates the registry when no `--name` passed

Original spec mentions this in step 8. Make it concrete:

- `bash doctor.sh` (no flag) → reads `instances.json`, prints a
  one-line summary per instance, then runs the full per-instance
  checks for the *first* / *only* instance (or, if multiple, asks
  which to deep-dive on).
- `bash doctor.sh --name <name>` → full per-instance checks for the
  named instance.
- `bash doctor.sh --all` → full per-instance checks for every
  instance.

### 7. Bot-token collision check (promote to acceptance criteria)

Already in "Notes / risks". Promote to explicit acceptance:

> [ ] Install with a `BOT_TOKEN` already used by another instance
> refuses with a clear error before touching state. The Telegram
> Bot API allows only one polling client per token; two instances
> sharing one token would silently fight each other.

Implementation: scan `~/.claudify/instances/*/channels/telegram/.env`
for the new `BOT_TOKEN`; refuse if found.

### 8. Forward `--name` through `update.sh` and `uninstall.sh`

Original spec says yes, just being explicit:
- `update.sh` accepts `--name <n>` and forwards to `install.sh
  --preserve-state --name <n>`
- `uninstall.sh` requires `--name <n>` OR `--all`; bare invocation
  is rejected with a list of instances and instructions

## Post-research delta — multi-client isolation (2026-05-07)

> Added 2026-05-07 after the multi-client security discussion. The
> earlier nested layout (`~/.claudify/instances/<name>/`) plus
> `ProtectHome=read-only + ReadWritePaths` was insufficient for a
> real multi-client deployment — same Linux user means same
> filesystem ownership, so a prompt-injected Client A bot could
> `cat ~/.claudify/instances/client-b/credentials.env` and exfiltrate.
> This delta replaces both the layout AND the security mechanism.
>
> Recorded as [ADR 0006](../../decisions/0006-multi-client-isolation.md).
> 3.6.2 (filesystem write-restriction) is updated to match.

### A. Switch to flat top-level layout

Per-instance folders move to `~/.claudify-<name>/` (top-level, flat),
not `~/.claudify/instances/<name>/` (nested). Plus a side-car
registry at `~/.claudify-registry.json`.

```
~/                                       ← operator's home
├── .claudify-registry.json              ← side-car registry (was instances.json under one root)
├── .claudify-client-a/                  ← Client A's instance, fully self-contained
│   ├── claudify.json                    ← per-instance manifest
│   ├── credentials.env                  ← Client A's OAuth (chmod 600)
│   ├── workspace/CLAUDE.md              ← Client A's persona
│   ├── channels/telegram/.env           ← Client A's bot token
│   ├── channels/telegram/access.json    ← Client A's allowlist
│   ├── mcps/, skills/, hooks/           ← Phase 4 extensions, per-instance
│   ├── data/                            ← per-skill SQLite, persona.db, etc.
│   └── claude/                          ← NEW: per-instance Claude state
│       ├── settings.json
│       ├── plugins/
│       └── (anything else CLAUDE_CONFIG_DIR captures)
├── .claudify-client-b/                  ← Client B's instance, identical shape
└── .claudify-client-c/                  ← etc.
```

**Why flat:** psychological isolation maps directly to filesystem
isolation. Each client is a top-level folder; `BindPaths=` (next
section) makes one folder visible to one service and no others.

**Naming:** every install requires an explicit `--name` value — no
auto-renaming. First-time install with no flag uses
`--name default`, producing `~/.claudify-default/`. The single-bot
operator never thinks about the suffix; multi-client operators name
their instances on purpose (`--name client-a`, `--name client-b`).

### B. Mount-namespace isolation via systemd

Earlier 3.6.2 plan: `ProtectHome=read-only + ReadWritePaths=…`.
**Insufficient** — restricts writes only; reads of other instances'
folders still succeed.

Replacement: `ProtectHome=tmpfs + BindPaths + BindReadOnlyPaths`.
This gives each unit a private filesystem view where other
instances' folders **don't exist** at all.

```ini
# In Client A's unit (claudify-client-a.service):
ProtectHome=tmpfs
BindPaths=%h/.claudify-client-a
BindReadOnlyPaths=%h/.npm-global %h/.bun
```

What Client A's bot sees inside its mount namespace:
```
$HOME/                        ← tmpfs, otherwise empty
├── .claudify-client-a/       ← read-write (their own data + claude state)
├── .npm-global/              ← read-only (the claude binary)
└── .bun/                     ← read-only (the bun runtime)
```

What it **cannot see** (paths don't exist in its namespace):
- `~/.claudify-client-b/` ← invisible
- `~/.claudify-client-c/` ← invisible
- `~/.ssh/`, `~/.bashrc`, anything else under `$HOME`

This is **kernel-enforced** isolation, not policy-enforced. A
prompt-injected bot asking *"read every credentials file in /home"*
finds nothing because /home is essentially empty in its view.

### C. Per-instance Claude state via `CLAUDE_CONFIG_DIR`

Claude Code reads `~/.claude/` and `~/.claude.json` (project trust,
plugins, settings) by default — these are user-wide. **For
multi-client isolation, every instance needs its own.**

Anthropic supports this via the `CLAUDE_CONFIG_DIR` env var
(confirmed in [their IAM docs](https://docs.claude.com/en/docs/claude-code/iam)).
Each instance's systemd unit sets:

```ini
Environment=CLAUDE_CONFIG_DIR=%h/.claudify-client-a/claude
```

Now per-instance: `settings.json`, plugins cache, project-trust
state. Client A's plugins are invisible to Client B even if they're
"the same" plugin technically.

`engine_seed_state` in `lib/engines/claude-code.sh` is updated to
write into `${CLAUDIFY_INSTANCE_DIR}/claude/` (resolved via
`CLAUDE_CONFIG_DIR`) rather than `~/.claude/`.

### D. Registry as a side-car file

The registry file moves:

| Before (nested) | After (flat) |
|---|---|
| `~/.claudify/instances.json` | `~/.claudify-registry.json` |

Schema is unchanged — same `{version, instances: {<name>: {…}}}`
shape. Each install appends/updates its own entry. Each
uninstall removes its entry.

`lib/manifest.sh` `_registry_path` is the only function that
changes. The 9 other manifest helpers are path-resolved via
`_instance_manifest_path` (which now returns
`$HOME/.claudify-<name>/claudify.json` instead of
`$HOME/.claudify/instances/<name>/claudify.json`).

### E. Path naming convention summary

| Item | Path |
|---|---|

| Per-instance hooks | `~/.claudify-<name>/hooks/<hook>/` |
| Per-instance data | `~/.claudify-<name>/data/<skill-id>/` |

## Post-experiment update — 2026-05-10 — namespace isolation dropped

> **Critical finding from Station11 testing.** The
> `ProtectHome=tmpfs + BindPaths + BindReadOnlyPaths` directives
> from the previous delta were tested on Station11 (Ubuntu 24.04)
> and confirmed to **silently no-op in user-mode systemd**.
>
> **Root cause:** `kernel.apparmor_restrict_unprivileged_userns=1`
> (Ubuntu 24.04 default since LTS launch). Even with
> `PrivateUsers=yes` added, the unit's mount namespace was never
> created — `/proc/<pid>/ns/mnt` showed the same ID as the
> operator's shell. The bot retained full visibility of `~/.ssh`,
> `~/.bashrc`, and other instances' folders.
>
> **What works:** the directives parse without error;
> `systemctl show` reports them as set; nothing in the journal
> warns about the silent fallback. So the unit *looked* correct
> but provided **zero** isolation. Shipping it as-is would have
> been false advertising.

### What changes

**Drop from this task:**
- `ProtectHome=tmpfs` directive
- `BindPaths=%h/.claudify-<name>` directive
- `BindReadOnlyPaths=%h/.npm-global %h/.bun` directive
- The "two instances side-by-side namespace verification" acceptance
  criterion (it's not testable — there's no namespace to verify)

**Keep — these all work fine and are still in scope for 3.4.5:**
- Flat layout `~/.claudify-<name>/`
- `--name` flag + multi-instance support
- Per-instance systemd unit `claudify-<name>.service`
- Side-car registry `~/.claudify-registry.json`
- Per-instance Claude state via `CLAUDE_CONFIG_DIR`
- Bot-token collision check across instances
- `WORKSPACE` prompt removed (instance name covers it)
- doctor.sh iterates the registry
- Bot-token / state per instance
- Manifest-driven engine + channel registration

### What 3.4.5 actually delivers (honest)

**Multi-instance bash install — organisational, not isolation.** The
operator can run multiple bots on one host with separate folders,
units, manifests, and Telegram bots. They share the operator's
filesystem ownership; nothing kernel-enforced separates them.

**This is fine for:**
- One operator running 2-3 personal/persona bots they trust themselves
- Development / testing scenarios

**This is NOT fine for:**
- Hosting bots for paying clients (they don't trust each other and
  you don't trust them)
- Any genuinely multi-tenant scenario

For real isolation: see [3.4.9 Containerize Claudify](3.4.9-containerize.md).
Each container is its own namespace + cgroup, kernel-enforced.

### What replaces "Tier-2 isolation" in the security plan

| Original plan | New plan |
|---|---|
| 3.6.2 used `ProtectHome=tmpfs + BindPaths` for cross-instance isolation | 3.6.2 spec is updated: this is now the **anti-foot-gun** task — strip the broken directives so the unit doesn't *claim* isolation it doesn't deliver |
| ADR 0006 prescribed mount-namespace isolation | ADR 0006 gets a 2026-05-10 appendix documenting the failure; the canonical answer for multi-tenant becomes containers (3.4.9) |
| `nsenter` verification step in 3.6.2 | Removed |
| Mount-namespace-related directives in service.sh | Removed |

Tier-1 hardening from 3.6.1 (`NoNewPrivileges`, `PrivateTmp`,
`Protect*`, resource caps) **stays** — those work on Ubuntu 24.04
and protect each instance from itself (kernel-state, fork-bombs,
memory exhaustion). Just doesn't isolate cross-instance.

### Dual delivery is the answer

Two paths, well-defined uses:

| Path | Isolation | Use |
|---|---|---|
| Bash install (3.4.5) | Tier-1 hardening only — protects host from a misbehaving bot but not bots from each other | Solo, 1-3 bots, operator trusts their skills |
| Container (3.4.9) | Kernel-enforced via container namespace + cgroups | Hosted multi-tenant, codaki.com, paranoid solo |

3.4.5 ships the bash path honestly. 3.4.9 ships the container path
for anyone who needs real isolation.
