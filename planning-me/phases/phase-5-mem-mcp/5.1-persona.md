# Task 5.1 вЂ” `persona.db` + `lib/persona.sh` + auto-render via `engine_apply_persona`

**Status:** вЏі pending

## Goal

A shared `persona.db` exists at
`~/.claudify-<name>/data/_persona/persona.db` with a `facts` table.
Bootstrapped from operator answers in onboarding (name, timezone,
language, working hours). On every persona update, the rendered
markdown block is pushed into the engine's surface via
`engine_apply_persona` (3.4.5.2's contract function вЂ” already shipped).

## Why

The persona is the operator's identity from the model's POV: name,
timezone, working hours, language, conventions. Today the persona
lives only inside the seeded `CLAUDE.md` and the operator edits it
by hand. After 5.1, it lives as structured rows in `persona.db`,
auto-rendered into a marker-bracketed block in `CLAUDE.md`. The
operator can change facts via `default remember` (5.1.1) and the
CLAUDE.md block reflects it within seconds.

## Schema

Per `.planning/research/memory.md` Part C:

```sql
CREATE TABLE facts (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  source TEXT,
  sensitive BOOLEAN DEFAULT 0,
  updated_at TEXT NOT NULL
);
```

## Steps

1. Write `lib/persona.sh` with the 5 public functions.
2. `persona_init` opens (or creates) `data/_persona/persona.db` via sqlite3, applies schema, then `INSERT OR IGNORE` for each onboarding fact.
3. `persona_set` writes via SQL UPSERT, then calls `engine_apply_persona "$(persona_render)"` to refresh CLAUDE.md.
4. `persona_render` outputs the marker-bracketed markdown block.
5. Wire into `install.sh::main()` вЂ” `persona_init` after substrate creation, immediately followed by `engine_apply_persona "$(persona_render)"`.
6. Write `tests/bash/persona.bats`.
7. `bash test.sh` вЂ” bats green.
8. **Station11 round-trip:** clean install; `~/.claudify-<n>/data/_persona/persona.db` has the 4 bootstrap rows; `workspace/CLAUDE.md` has the marker-bracketed block at the top; `default remember` (after 5.1.1) updates both.
9. CHANGELOG: `### Added вЂ” persona.db with facts table at data/_persona/. Auto-renders into CLAUDE.md via engine_apply_persona on every persona_set.`

## Out of scope

- The `default remember` operator command (that's 5.1.1)
- Auto-extraction of facts from messages (mem0 style) вЂ” explicitly anti-vision
- Sensitive-fact redaction during render (sensitive facts ARE rendered into CLAUDE.md by design вЂ” the model needs them; the `sensitive` flag is for audit/UI, not censorship)

## Notes / risks

- **Two write paths must stay in sync:** the lib helper `persona_set` AND the MCP tool `persona.set` (5.0a). Both write to the same DB; both call the same render path. The MCP version is gated by manifest declaration (only the persona skill can write `persona.db`).
- **`engine_apply_persona` is the engine-agnostic boundary:** persona.sh never knows what file it ends up writing to. The Claude Code adapter writes `workspace/CLAUDE.md`; a future Gemini adapter writes its equivalent. persona.sh's job is to render + hand off.
- **Bootstrap from onboarding:** `_collect_inputs_fresh` (lib/onboarding.sh) already collects name/timezone/language equivalents implicitly. 5.1 wires those answers into `persona_init` as the seed rows.
