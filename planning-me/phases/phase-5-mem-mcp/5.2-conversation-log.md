# Task 5.2 — Conversation log + FTS5 + `<private>` filter

**Status:** ⏳ pending
**Estimated effort:** ~1 hr
**Depends on:** 5.0a (MCP exposes search), 5.0b (MCP wired into engine)
**Blocks:** Phase 6.x skills that want to query past conversations

## Goal

Every message exchanged via Telegram (or any future channel) gets
logged to `data/_conversations/messages.db`. The MCP's `memory.search`
and `memory.recent` tools (from 5.0a) query against this DB.
`<private>...</private>` spans are stripped before insert.

## Why

Memory of past conversations is the difference between "an LLM
behind Telegram" and "an assistant that knows what we talked about
yesterday." Stored locally, queryable via FTS5, with explicit privacy
controls — `<private>` for one-off redactions, `<no-log>` for entire
messages.

## Scope

### Files to create

- `lib/conversation-hook.sh` (~80 lines) — opens the DB in WAL mode, strips `<private>` spans, inserts inbound + outbound rows. Idempotent / safe to be called concurrently.
- `tests/bash/conversation-hook.bats` (~60 lines) — covers redaction + insert + private-span stripping + no-log skip

### Files to modify

- `data/_conversations/messages.db` — schema per `memory.md` Part C, including the `messages_fts` FTS5 virtual table. Created on first install or by the MCP at startup.
- Engine adapter (`lib/engines/claude-code.sh`) — register hook on `PostToolUse` for Telegram send/receive
- `install.sh` — call `conversation_hook_setup` so first install creates the schema
- `CHANGELOG.md` — `### Added` entry

## Schema (from `memory.md` Part C)

```sql
CREATE TABLE messages (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  ts        TEXT NOT NULL,             -- ISO 8601 UTC
  direction TEXT NOT NULL,             -- 'in' | 'out'
  channel   TEXT NOT NULL,             -- 'telegram' | 'web' | ...
  user_id   TEXT,
  text      TEXT NOT NULL              -- after <private> stripping
);

CREATE VIRTUAL TABLE messages_fts USING fts5(
  text, content='messages', content_rowid='id'
);

CREATE TRIGGER messages_ai AFTER INSERT ON messages BEGIN
  INSERT INTO messages_fts(rowid, text) VALUES (new.id, new.text);
END;
```

## Privacy controls

- **`<private>secret</private>`** — span is stripped before insert. The model sees it; the log doesn't.
- **`<no-log>...</no-log>`** — wraps an entire message; that message is skipped (not inserted at all). Per-message opt-out.

## Steps

1. Write `lib/conversation-hook.sh` with one main function: `conversation_log <direction> <channel> <user_id> <text>`.
2. The function: opens the DB (WAL), strips `<private>` spans via sed/awk, checks for `<no-log>` outer wrap, inserts if not skipped.
3. Bootstrap the schema: `conversation_hook_setup` (called from install.sh) opens the DB and applies the schema if not already there.
4. Wire the hook into the engine adapter — Claude Code's `PostToolUse` hook fires after each Telegram tool call; from inside the hook we call `conversation_log out telegram ...` for outbound and `conversation_log in telegram ...` for inbound.
5. Write `tests/bash/conversation-hook.bats`.
6. `bash test.sh` — bats green.
7. **Station11 round-trip:** send a few Telegram messages (one normal, one with `<private>`, one wrapped in `<no-log>`); verify DB rows match expectations.
8. CHANGELOG: `### Added — conversation log at data/_conversations/messages.db (FTS5-queryable via MCP). <private> spans stripped before insert; <no-log> wraps skip the entire message.`

## Out of scope

- Encryption-at-rest of messages.db (Phase 7+ if needed; OS-level disk encryption covers most threat models)
- Compression / summarisation of old rows (Phase 7+, triggered by token-cost concern in real use)
- Cross-channel federation (single channel: Telegram, today)
- Hook installation for non-Claude engines (engine adapter handles its own hook mechanism; the contract is "call conversation_log when a message lands")

## Notes / risks

- **Hook reliability is critical.** A missed hook = a lost message in the log = the bot can't recall it later. Use `PostToolUse` (fires after tool succeeds), not pre-hook. Test for the hook being called on every Telegram tool invocation.
- **`<private>` parsing edge cases:** nested spans, malformed tags, multi-line spans. The redaction must handle all three. Be liberal in stripping (over-redaction is safer than under-redaction).
- **DB grows unbounded.** No retention policy in this task. Phase 7 may add row-aging if real-world use shows token-cost issues. For now, ~1 row per message at ~200 bytes — even at 1000 msgs/day, that's 73 MB/year. Reasonable.
- **FTS5 trigger:** the AFTER INSERT trigger automatically populates the FTS5 mirror. Don't forget the matching AFTER DELETE / UPDATE triggers if we ever support message editing.
