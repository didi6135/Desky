# Task 5.0a — `claudify-memory` MCP server

**Status:** ⏳ pending
**Estimated effort:** ~3 hr
**Depends on:** 3.4.5.1 (data substrate — done), 3.4.5.2 (engine contract stub — done), Phase 3 closure recommended (3.5 backup, Phase 4 security)

**Research:** [`.planning/research/memory.md`](../../research/memory.md) Part C

## Goal

A working MCP server in `src/mcp/memory/` that exposes the 9 tools
defined in `memory.md` Part C: `memory.list`, `memory.read`,
`memory.write`, `memory.append`, `memory.delete`, `memory.search`,
`memory.recent`, `persona.get`, `persona.set`. Backed by the file
substrate (`data/_memories/*`) + SQLite stores
(`data/_persona/persona.db`, `data/_conversations/messages.db`).
All SQLite opens in WAL mode. All writes recorded to
`data/_audit/writes.log`.

## Why

This is the substrate of Phase 5. Every subsequent task in Phase 5
(persona, conversation log) and every Phase 6 skill that touches
memory calls into this MCP. By building it first, we make every
downstream task's surface area smaller — they call tools, they don't
re-implement storage.

## Scope

### Files to create

- `src/mcp/memory/index.ts` — entrypoint, MCP boilerplate, tool registration (~80 lines)
- `src/mcp/memory/files.ts` — 5 file-op tools confined to `/memories/` with path-traversal protection (~120 lines)
- `src/mcp/memory/search.ts` — FTS5 wrappers + `memory.recent` (~100 lines)
- `src/mcp/memory/persona.ts` — `persona.get` + gated `persona.set` (~80 lines)
- `src/mcp/memory/audit.ts` — append-only audit log helper (~40 lines)
- `src/mcp/memory/db.ts` — SQLite open helpers, schema migrations, WAL mode (~60 lines)
- `tests/ts/mcp-memory.test.ts` — Bun tests with real SQLite + tmp filesystem (~150 lines)

### Files to modify

- `src/package.json` — pin MCP SDK + `@types/bun` versions
- `build.sh` — bundle the MCP into `dist/` (or build separately as part of install)

## Notes / risks

- **Engine-agnostic by construction:** the MCP server doesn't know which model is on the other side. Future Gemini adapter reuses it as-is.
- **All SQLite opens in WAL mode** so concurrent reads + a single writer never block. Per `memory.md` Part C.
- **Path-traversal guard is the security-critical bit.** Resolve paths via `path.resolve` and verify the prefix is within the allowed root before any open. Test explicitly.
- **`persona.set` gating:** consult the per-instance `claudify.json` for which skill is allowed to write to `persona.db`. Reuses the `manifest_get_skill_memory` helper from 3.4.5.1 — but exposed via the MCP, called from TypeScript, so we'll need a small bridge or duplicate the lookup logic.
