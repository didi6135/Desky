# Task 6.4 — `<instance> skill new` template generator

**Status:** ⏳ pending
**Estimated effort:** ~1 hr
**Depends on:** 3.4.6 (personal cmd wrapper), 6.3 (skill subcommand exists)
**Blocks:** none — quality-of-life for skill authors

## Goal

```
$ client-a skill new my-skill
✓ wrote skills/my-skill/SKILL.md
✓ created skills/my-skill/{scripts,references,assets}/
✓ skeleton ready to fill in
```

Generated skill matches `docs/skills.md §5` template exactly so a
brand-new author can be productive in 60 seconds.

## Why

Without 6.4, skill authors copy-paste from `docs/skills.md` and miss
sections, mis-format the frontmatter, or forget the Gotchas block.
A `skill new` generator codifies the convention. Plus: it's the
on-ramp from "operator" to "skill author" — the moment someone goes
from "I install skills" to "I write skills."

## Scope

### Files to create

- `lib/skill-new.sh` — extracted into its own module since `lib/skill-install.sh` (6.3) is already a candidate to split

### Files to modify

- The personal cmd `skill` subcommand (6.3) — add `new` action
- `CHANGELOG.md` — `### Added` entry

## Generated layout

```
~/.claudify-<n>/skills/<id>/
├── SKILL.md                  ← from template, frontmatter pre-filled
├── scripts/                  ← empty, ready for add.sh / list.sh / etc.
├── references/               ← empty, ready for cheat-sheets
└── assets/                   ← empty, ready for fixed files (logos, prompts)
```

## Template content

`SKILL.md` body has these sections, placeholder-filled:

```markdown
---
id: <id>
version: 0.1.0
memory:
  writes:  # e.g. "<id>.db"
  reads: []  # e.g. ["persona.db"]
---

# <id>

<!-- One-paragraph description: what this skill does, when to use it. -->

## When to use

<!-- Concrete operator-facing trigger phrases or situations -->

## Procedure

<!-- Step-by-step what the skill does when invoked -->

## Gotchas

<!-- Non-obvious things that bite authors: timezone, concurrency, edge cases -->

## Files in this skill

<!-- List + describe everything under this dir -->
```

## Steps

1. Write `lib/skill-new.sh` with `skill_new <id>` that creates the directory tree and templated SKILL.md.
2. Wire into the personal cmd's `skill new` action.
3. Validate the generated skeleton passes the 6.3 install-validation gate (sanity: don't generate something that can't be installed).
4. Tests: generate to tmp dir + diff against expected layout.
5. CHANGELOG.

## Acceptance criteria

- [ ] `client-a skill new my-skill` produces a working skeleton
- [ ] Skeleton passes the manifest validation gate from 6.3
- [ ] `SKILL.md` frontmatter has `id`, `version: 0.1.0`, empty-but-present `memory.writes` + `memory.reads`
- [ ] Body has the 4 standard sections: When to use / Procedure / Gotchas / Files in this skill
- [ ] All three subdirs (`scripts/`, `references/`, `assets/`) exist as empty dirs

## Test plan

- **Local (bats):** generate to tmp dir; diff against expected files
- **Station11:** generate a skill; immediately try `skill install ./my-skill` to validate the skeleton round-trips

## Out of scope

- Choosing initial skill type (cron / hook / on-demand) — operator fills it in
- Pulling existing skill via `--from-template <name>` — Phase 7+
- Generating language-specific scaffolds (TS / Python) — bash-first, others later

## Notes / risks

- **Template drift:** if `docs/skills.md §5` changes, this template should follow. Cheapest mitigation: have 6.4's tests assert the generated file matches the docs example (e.g. `diff <(skill_new test) docs/skills.md` extracted).
- **Auto-fill `memory.writes`:** leave it blank, let the author decide. Forcing `<id>.db` would couple the convention; some skills are read-only.
- **Idempotency:** if the dir already exists, refuse rather than overwrite — operator data could live there.
