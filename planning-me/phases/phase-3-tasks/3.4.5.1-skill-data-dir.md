# Task 3.4.5.1 — Skill data dir + `lib/memory.sh` + `${CLAUDIFY_SKILL_DATA}`

**Status:** ✅ DONE 2026-05-12
**Estimated effort:** ~45 min
**Depends on:** 3.4.5 (multi-instance layout — flat `~/.claudify-<name>/` is the parent for `data/<id>/`)
**Blocks:** Phase 4 entries that ship real skills

**Path correction vs. spec:** the spec below was written assuming the
nested layout `~/.claudify/instances/<n>/data/`. The actual layout
shipped is the flat per-ADR-0006 form: `~/.claudify-<n>/data/`.
Implementation uses `$CLAUDIFY_INSTANCE_DIR/data` which resolves
correctly under either layout — the helper code is unchanged from
this spec, only the example paths in the prose differ.

## Goal

Stand up the **storage substrate** for skills before any skill is
written. Two pieces:

1. `~/.claudify/instances/<name>/data/<skill-id>/` — per-skill data
   directory. Survives skill upgrades (skill code dir gets replaced;
   data dir persists).
2. `lib/memory.sh` — small helper module that resolves the data dir
   for a given skill ID, exposes the path as the `${CLAUDIFY_SKILL_DATA}`
   environment variable when the skill runs, and (optionally)
   enforces the manifest's `memory.writes` / `memory.reads`
   declarations.

After this task, a skill author can write `${CLAUDIFY_SKILL_DATA}/foo.db`
in their script and trust the path will exist, be private to their
skill, and survive upgrades — matching Anthropic's
`${CLAUDE_PLUGIN_DATA}` convention so cross-ecosystem skills work.

## Why

`docs/skills.md §10` documents the data-dir contract. Today there's
no place for that data to live. Future skills (reminders, expense
tracking, persona memory, conversation log) all need a stable
per-skill storage path, isolated from other skills' data, surviving
`update.sh` and `--reset-config`.

This is also what makes the **per-skill SQLite isolation** work
without a heavyweight permission system: file-level isolation. A
`reminders` skill that opens `${CLAUDIFY_SKILL_DATA}/reminders.db`
literally cannot see a `calendar` skill's database — different
paths, different files. No wrapper layer needed for accident
prevention.

The storage primitive is foundational; ship it before any skill
needs it.

## `lib/memory.sh` — public API

| Function | Args | Behavior |
|---|---|---|
| `memory_dir` | `<skill-id>` | Echo the absolute path of the skill's data dir; `mkdir -p` it if missing; chmod 700. |
| `memory_path` | `<skill-id> <filename>` | Echo `<memory_dir>/<filename>`; convenience for SQLite-style usage. |
| `memory_assert_write` | `<skill-id> <db-name>` | Read the manifest entry for the skill; fail if `memory.writes` doesn't include `<db-name>`. Skill code calls this before opening for write. |
| `memory_assert_read` | `<skill-id> <db-name>` | Same, for reads. (Reads from another skill's dir require `memory.reads` to list it.) |
| `memory_export_env` | `<skill-id>` | Sets `CLAUDIFY_SKILL_DATA="$(memory_dir <skill-id>)"`; called by the agent right before invoking a skill. |

The assert helpers fail with a clear "skill X tried to write Y but
manifest only allows Z" message — accident prevention, not
malicious-actor defence.

## Steps

1. Write `lib/memory.sh` with the 5 public functions above + private
   resolver `_memory_root` returning `${CLAUDIFY_INSTANCE_DIR}/data`.
2. Update `install.sh` to `mkdir -p` the data root during install
   (after `write_configs`, alongside `seed_persona`).
3. Extend `lib/manifest.sh`:
   - `manifest_set_skill <instance> <skill-id> [writes-json] [reads-json]`
     — writes a skill entry with optional memory declaration
   - `manifest_get_skill_memory <instance> <skill-id>` — returns the
     `memory` object as JSON, or empty if not declared
4. Update `build.sh` MODULES (after `manifest.sh`).
5. `bash test.sh` — bats tests pass.
6. **Station11 round-trip:**
   - clean install
   - `ls -la ~/.claudify/instances/default/data/` exists, mode 700
   - `bash -c 'source lib/memory.sh; memory_dir test-skill'` outputs
     the right path; mkdir actually created it; perms are 700
7. CHANGELOG entry.

## Notes / risks

- The manifest gains a new optional field. Existing manifests
  (without `skills`) must still parse — `manifest_get_skill_memory`
  returns empty for absent entries, never errors.
- chmod 700 on the data dir is per-skill privacy; the instance dir
  itself stays 755 so doctor / uninstall can navigate.
- We do **not** introduce a daemon / broker process. File-level
  isolation is enough for the single-user trust model.
- `CLAUDIFY_SKILL_DATA` matches Anthropic's `CLAUDE_PLUGIN_DATA`
  intentionally — same shape, different name. Skills authored for
  Claude Code's plugin model can use either.
