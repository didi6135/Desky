# Task 3.4.4 — Manifest files (registry + per-instance)

**Status:** ✅ DONE 2026-05-04
**Estimated effort:** ~1 hr
**Depends on:** 3.4.2 (modules in place)
**Blocks:** 3.4.5 (multi-instance reads/writes the registry)

## Goal

Two new JSON files become the *single source of truth* for "what's
installed":
- `~/.claudify/instances.json` — top-level registry of all instances
- `~/.claudify/instances/<name>/claudify.json` — per-instance manifest

Add a `lib/manifest.sh` module with read/write helpers. `install.sh`
writes both at the end of the install. `doctor.sh`, `update.sh`,
`uninstall.sh`, `backup.sh` all read them as the source of truth for
"what state exists".

## Why

`docs/architecture.md §5` defines the schema. Without manifests, every
script has to guess what's installed by inspecting filesystem layout
— fragile, can't represent versions or "this MCP is enabled but not
configured." Manifests centralize that.

## Scope

## Schema (per `architecture.md §5`)

### `~/.claudify/instances.json`
```json
{
  "version": 1,
  "instances": {
    "default": {
      "created_at": "2026-04-28T10:00:00Z",
      "engine": "claude-code",
      "service": "claudify-default",
      "personal_cmd": "default"
    }
  }
}
```

### `~/.claudify/instances/<name>/claudify.json`
```json
{
  "version": 1,
  "name": "default",
  "created_at": "2026-04-28T10:00:00Z",
  "claudify_version": "0.1.0",
  "engine": "claude-code",
  "engine_version": "2.1.119",
  "channels": {
    "telegram": { "installed_at": "...", "enabled": true, "version": "0.0.6" }
  },
  "mcps": {},
  "skills": [],
  "hooks": []
}
```

## `lib/manifest.sh` — public functions

| Function | Args | Behavior |
|---|---|---|
| `manifest_init_registry` | — | Create `~/.claudify/instances.json` if missing (idempotent) |
| `manifest_register_instance` | `<name>` | Add or update the instance's entry in the registry |
| `manifest_unregister_instance` | `<name>` | Remove from registry (used by uninstall) |
| `manifest_list_instances` | — | Echo each instance name on its own line |
| `manifest_get_instance` | `<name>` | Print the registry entry as JSON |
| `manifest_init_instance` | `<name>` | Create `claudify.json` under the instance dir if missing |
| `manifest_set_channel` | `<name> <channel> <version>` | Add/update channel entry in instance manifest |
| `manifest_set_mcp` | `<name> <mcp> <version>` | Same for MCPs |
| `manifest_read_field` | `<name> <jq-path>` | Read one field via jq |
| `manifest_atomic_write` | `<file> <new-contents>` | Helper: write to `.tmp`, then mv (atomic) |

All writes use `jq` + the atomic write helper. Reads use `jq -r`. Fail
loudly if `jq` isn't installed (preflight already checks this).

## Steps

1. Write `lib/manifest.sh` with the 10 functions above
2. Use `jq` for every read/write — no string manipulation
3. Atomic-write helper: `mktemp` → write → `mv` (preserves perms)

## Notes / risks

- Atomic write is critical — partial JSON corruption is the worst-case outcome for a manifest. Test the fail path (kill mid-write, confirm original is intact).
- `jq -e` failures need clear error messages; manifest corruption deserves "your install state may be corrupt — restore from backup or re-install" guidance, not a silent crash.
- Read carefully: `jq` defaults to non-atomic in-place edits. Always use the temp-file pattern.
