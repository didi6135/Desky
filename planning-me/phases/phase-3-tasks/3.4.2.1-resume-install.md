# Task 3.4.2.1 — Resume interrupted install

**Status:** ✅ DONE 2026-05-04
**Estimated effort:** ~30 min
**Depends on:** 3.4.2 (needs `lib/onboarding.sh` to exist)
**Blocks:** —

## Goal

If a user pastes their bot token + Telegram user ID and then Ctrl-Cs
(network drop, second-thoughts, anything), re-running `install.sh`
picks up exactly where it left off — no re-paste, no flag to remember.

## Why

Today, `BOT_TOKEN` / `TG_USER_ID` / `WORKSPACE` only hit disk during
`write_configs` (step 7 of ~10). If the user Ctrl-Cs anywhere between
`collect_inputs` (step 4) and `write_configs`, those inputs are lost.
First-paste UX is fine; second-paste UX is annoying.

Goal per the owner: *"in the end I want that it's will be very easy
to deal with that — and if it's become heavy so I miss the all point
of all of that."*

So: no new flags, no `--resume`, no "Would you like to resume?"
prompt. The behavior is invisible until it's needed.

## Steps

1. Add `_write_partial_state` helper in `onboarding.sh`: writes
   `BOT_TOKEN=<v>\nTG_USER_ID=<v>\nWORKSPACE=<v>\n` to
   `$CLAUDIFY_ROOT/.install-partial` with `chmod 600`. Bot token is in
   there — the file MUST be 600 from the moment it exists.
2. Add `_load_partial_state` helper: if `-s "$CLAUDIFY_ROOT/.install-partial"`
   AND none of the three vars is already set in env, source it via
   `set -a; . file; set +a`. Print one green `✓ resumed from previous
   attempt` line.
3. Wire in `collect_inputs`: try `_load_partial_state` first. If it
   loaded everything, return early. Otherwise fall through to
   `_collect_inputs_preserved` or `_collect_inputs_fresh`.
4. End of `_collect_inputs_fresh`: call `_write_partial_state` (so
   inputs hit disk *before* any heavy work — the whole point).
5. `intro()` (in `onboarding.sh`): append the "Safe to Ctrl-C" line.
6. `final_summary()` success branch: `rm -f "$CLAUDIFY_ROOT/.install-partial"`.
7. `parse_args` (in `args.sh`): when `--reset-config` is set, append a
   note that the partial state will be wiped (it's wiped by
   `_load_partial_state` checking `RESET_CONFIG` and skipping).
   Simpler: just `rm -f` the partial in `_load_partial_state` when
   `RESET_CONFIG=1`.
8. README "Install" section: add the two-line note after the curl
   command — "Safe to Ctrl-C; re-run picks up where you stopped."
9. CHANGELOG `### Added` entry under Unreleased.
10. `bash -n` every touched file.
11. `bash build.sh` — `dist/install.sh` rebuilds clean.
12. **Station11 round-trip:**
    - scrub install state
    - paste BOT_TOKEN + TG_USER_ID, Ctrl-C between `collect_inputs`
      and `install_claude`
    - `cat ~/.claudify/.install-partial` — confirm chmod 600 + 3 vars
    - re-run install — confirm `✓ resumed from previous attempt` and
      no re-prompts
    - finish install — confirm `.install-partial` is gone after
      `final_summary`

## Out of scope

- Resuming from steps **after** `write_configs` — that's already
  handled by the existing `--preserve-state` flag (and the partial
  state file is redundant once `telegram/.env` exists).
- Multi-instance partial state (`instances/<name>/.install-partial`) —
  3.4.5 will move it under the instance dir as part of the layout
  migration.
- Auto-detecting partial state in `update.sh` — `update.sh` already
  uses `--preserve-state`; partial state is for the initial install
  only.

## Notes / risks

- **File location:** `$CLAUDIFY_ROOT/.install-partial` (i.e.,
  `~/.claudify/.install-partial`). The dir might not exist when
  `_write_partial_state` runs — `mkdir -p "$CLAUDIFY_ROOT"` first.
- **Permissions:** the file holds the bot token. Set `umask 077` then
  `chmod 600` after write to be belt-and-suspenders. Same pattern as
  `credentials.env`.
- **Cleanup on failure:** if install fails *after* `_write_partial_state`
  but *before* `final_summary`, the partial sticks around. That's
  intentional — it's exactly what enables resume.
- **Conflict with `--preserve-state`:** if `.install-partial` AND
  `~/.claudify/telegram/.env` both exist, prefer `.env` (it's the
  authoritative state). `_load_partial_state` returns early if
  `PRESERVE_STATE=1`.
- **Shell-special chars in values:** bot tokens are
  `[0-9]+:[A-Za-z0-9_-]+` — safe in plain `KEY=value` lines. Don't
  quote on write (would mangle the value on re-source); the
  `set -a; . file; set +a` pattern handles unquoted values fine.
