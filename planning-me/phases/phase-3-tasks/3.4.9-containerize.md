# Task 3.4.9 — Containerize Claudify (dual delivery: bash install + Docker image)

**Status:** ⏳ pending
**Estimated effort:** ~7 hr (one focused day)
**Depends on:** 3.4.5 (multi-instance refactor must close), 3.4.8 (docs synced — README points at the right install)
**Blocks:** Phase 5+ — codaki.com hosted infrastructure (each client = one container)

## Goal

Add a second delivery shape to Claudify alongside the curl-bash install: a **Docker (Podman-compatible) image** that wraps the same code in a container. After this task:

| Audience | Install path | Where it runs |
|---|---|---|
| Solo user (own server, own bot) | `curl -fsSL .../install.sh \| bash` | Their host's user-mode systemd, exactly as today |
| Container user (own laptop testing, codaki.com hosting) | `docker run ghcr.io/didi6135/claudify:latest …` or `podman run …` | Inside a sealed container with kernel-level isolation |

**Single codebase, two delivery paths.** Same `lib/`, same engine adapter, same MCPs, same `docs/skills.md` contract for skill authors. The difference is purely how the bot is *wrapped* and *launched*.

## Why

### For solo users
Container delivery isn't strictly *necessary*. Curl-bash works. But it's a bonus: people who already use Docker can run Claudify like any other containerized app, with `docker logs`, `docker stats`, `docker restart` as their tooling.

### For codaki.com (hosted multi-tenant)
Container delivery is the **foundation**. Without it, codaki.com can't safely host 20+ untrusted clients on shared infrastructure. Containers provide the kernel-enforced isolation that the mount-namespace approach ([ADR 0006](../../decisions/0006-multi-client-isolation.md)) couldn't deliver on Ubuntu 24.04 (AppArmor blocks unprivileged user-namespace mount operations). The container IS the namespace; the host kernel enforces it.

### For local development
A reproducible Docker build means each commit can be smoke-tested without polluting the developer's host. `bash test-container.sh` becomes part of the inner CI loop alongside `bash test.sh`.

## The Dockerfile

Outline. Final tuned in implementation.

```dockerfile
# Base — Ubuntu 24.04 to match Station11 + most cloud VMs
FROM ubuntu:24.04

# Pinned Node + system tools
RUN apt-get update && apt-get install -y --no-install-recommends \
      curl ca-certificates jq util-linux nodejs npm tini \
    && rm -rf /var/lib/apt/lists/*

# Bun (claude plugins use it for MCP servers)
ENV BUN_INSTALL="/opt/bun"
RUN curl -fsSL https://bun.sh/install | bash
ENV PATH="${BUN_INSTALL}/bin:/opt/npm-global/bin:${PATH}"

# claude-code globally available
RUN npm config set prefix /opt/npm-global \
    && npm install -g @anthropic-ai/claude-code

# Claudify source — the lib/ + entrypoint
COPY lib/         /app/lib/
COPY install.sh   /app/install.sh
COPY build.sh     /app/build.sh

# State volume — the only writable path that survives container restart
VOLUME /state

# Container mode marker + state path
ENV CLAUDIFY_CONTAINERIZED=1
ENV CLAUDIFY_STATE_DIR=/state

# Run as non-root inside the container
RUN useradd -m -u 1000 claudify
USER claudify

ENTRYPOINT ["/usr/bin/tini", "--", "/app/lib/boot.sh"]
```

### Why these choices

- **Ubuntu 24.04 base** — matches Station11; same kernel + glibc behaviour; well-known.
- **`tini`** as PID 1 wrapper — handles signal forwarding + zombie reaping properly. Standard pattern for Node/Bun-based apps in containers.
- **Bun + claude-code baked into image** — clients shouldn't run `npm install` at container start (slow, fragile). Image rebuilds on Claudify version bumps.
- **Non-root user inside container** (`claudify`) — defense in depth even if a process escapes the container; lands on an unprivileged uid.
- **`/state` as the only volume** — everything mutable lives there; image itself is immutable.

### Image size estimate

| Layer | Size |
|---|---|
| Ubuntu 24.04 base | ~78 MB |
| apt deps + cleanup | ~120 MB |
| Bun | ~50 MB |

| Claudify code | ~1 MB |

## `lib/boot.sh` — the container entrypoint

Replaces systemd. Runs in this order:

1. **Validate required env vars** (`BOT_TOKEN`, `TG_USER_ID`, `CLAUDE_CODE_OAUTH_TOKEN`). Fail loud if missing.
2. **First-boot check** — if `/state/claudify.json` doesn't exist, this is a fresh container. Run the layout setup (mkdir tree, write configs). If it exists, skip — state survives across container restarts.
3. **Seed engine state** — `engine_seed_state` (writes `/state/claude/.claude.json` and `/state/claude/settings.json`).
4. **Install channel plugin** if not already (`engine_install_channel_plugin telegram`).
5. **Write the manifest** (`manifest_init_instance container`, `manifest_set_channel container telegram`).
6. **Exec claude** as the main process — `exec /usr/bin/script -qfec "claude --permission-mode bypassPermissions --channels plugin:telegram@..." /dev/null`.

The `exec` matters: claude becomes PID 1 (or PID 1's only child via tini). Container restart = claude exit. Docker handles the lifecycle.

### Pseudocode

```bash
#!/usr/bin/env bash
set -euo pipefail

: "${BOT_TOKEN:?BOT_TOKEN required (pass via -e BOT_TOKEN=...)}"
: "${TG_USER_ID:?TG_USER_ID required}"
: "${CLAUDE_CODE_OAUTH_TOKEN:?CLAUDE_CODE_OAUTH_TOKEN required}"

# Source the shared lib helpers (path-resolution, engine, manifest)
LIB_DIR=/app/lib
# shellcheck source=lib/ui.sh
source "$LIB_DIR/ui.sh"
# shellcheck source=lib/layout.sh
source "$LIB_DIR/layout.sh"   # detects CLAUDIFY_CONTAINERIZED=1, sets /state paths
# … etc.

# First-boot setup if /state isn't initialised yet
if [[ ! -s "$CLAUDIFY_INSTANCE_DIR/claudify.json" ]]; then
  mkdir -p "$CLAUDIFY_WORKSPACE" "$CLAUDIFY_TELEGRAM" "$CLAUDIFY_DATA" "$CLAUDIFY_CLAUDE_DIR"
  printf 'TELEGRAM_BOT_TOKEN=%s\n' "$BOT_TOKEN" > "$CLAUDIFY_TELEGRAM/.env"
  chmod 600 "$CLAUDIFY_TELEGRAM/.env"
  cat > "$CLAUDIFY_TELEGRAM/access.json" <<JSON
{"dmPolicy":"allowlist","allowFrom":["$TG_USER_ID"],"groups":{},"pending":{}}
JSON
  printf 'CLAUDE_CODE_OAUTH_TOKEN=%s\n' "$CLAUDE_CODE_OAUTH_TOKEN" > "$CREDS_FILE"
  chmod 600 "$CREDS_FILE"

  engine_seed_state "$CLAUDIFY_WORKSPACE"
  engine_install_channel_plugin telegram
  seed_persona

  manifest_init_registry
  manifest_register_instance container
  manifest_init_instance     container
  manifest_set_channel       container telegram ""

  ok "first-boot setup complete"
fi

# Smoke-only mode for CI: validate setup, exit without launching the bot
if [[ "${1:-}" == "--smoke-only" ]]; then
  ok "smoke test passed"
  exit 0
fi

# Run the bot. exec replaces this script with claude as PID 1 (under tini).
exec $(engine_run_args)
```

## Layout-detection in `lib/layout.sh`

Single conditional at the top:

```bash
INSTANCE_NAME="${INSTANCE_NAME:-default}"

claudify_init_layout() {
  if [[ "${CLAUDIFY_CONTAINERIZED:-0}" -eq 1 ]]; then
    # Container mode: single instance per container, fixed paths, /state volume
    INSTANCE_NAME="container"
    CLAUDIFY_INSTANCE_DIR="${CLAUDIFY_STATE_DIR:-/state}"
  else
    # Solo mode: per-instance flat layout
    CLAUDIFY_INSTANCE_DIR="$HOME/.claudify-$INSTANCE_NAME"
  fi
  CLAUDIFY_WORKSPACE="$CLAUDIFY_INSTANCE_DIR/workspace"
  CLAUDIFY_TELEGRAM="$CLAUDIFY_INSTANCE_DIR/channels/telegram"
  CLAUDIFY_DATA="$CLAUDIFY_INSTANCE_DIR/data"
  CLAUDIFY_CLAUDE_DIR="$CLAUDIFY_INSTANCE_DIR/claude"
  # … etc.
  CLAUDIFY_REGISTRY="${HOME:-/state}/.claudify-registry.json"
}

claudify_init_layout
```

That's the entire mode-switch. Everything downstream (configs.sh, engine adapter, manifest helpers) just reads `$CLAUDIFY_INSTANCE_DIR` — no other changes needed.

## `compose.yaml` — local dev

```yaml
services:
  claudify:
    build: .
    environment:
      BOT_TOKEN: ${BOT_TOKEN:?env var required}
      TG_USER_ID: ${TG_USER_ID:?env var required}
      CLAUDE_CODE_OAUTH_TOKEN: ${CLAUDE_CODE_OAUTH_TOKEN:?env var required}
    volumes:
      - claudify-state:/state
    restart: unless-stopped
    # Resource caps that codaki.com would enforce per-tenant
    mem_limit: 1g
    pids_limit: 200
    cpus: 1.0

volumes:
  claudify-state:
```

Workflow:

```bash
# Set creds in a local .env file (gitignored) or shell env
export BOT_TOKEN=… TG_USER_ID=… CLAUDE_CODE_OAUTH_TOKEN=…

# Build + run in one shot
docker compose up --build

# Watch logs
docker compose logs -f

# Test a fresh state
docker compose down -v && docker compose up --build
```

## Dual-delivery — what users see

### Path 1 — Solo (curl + bash, today's flow)

```bash
curl -fsSL https://raw.githubusercontent.com/didi6135/Claudify/main/dist/install.sh | bash
# … walks through Telegram setup, OAuth, etc.
# … bot runs as systemd user service
# … state at ~/.claudify-default/
```

**What changes for solo users:** nothing. Their experience is identical to today. The container mode is a parallel option, not a replacement.

### Path 2 — Container (Docker / Podman)

Step 1: get a long-lived OAuth token first (one time, on any machine):
```bash
claude setup-token
# → prints sk-ant-oat01-…
```

Step 2: run the container:
```bash
docker run -d --name my-claudify \
  --restart=unless-stopped \
  --memory=1g --pids-limit=200 \
  -e BOT_TOKEN="1234567890:ABCdef-…" \
  -e TG_USER_ID="7104012252" \
  -e CLAUDE_CODE_OAUTH_TOKEN="sk-ant-oat01-…" \
  -v claudify-state:/state \
  ghcr.io/didi6135/claudify:latest
```

Or with Podman (rootless):
```bash
podman run -d --name my-claudify \
  --memory=1g --pids-limit=200 \
  -e BOT_TOKEN=… -e TG_USER_ID=… -e CLAUDE_CODE_OAUTH_TOKEN=… \
  -v claudify-state:/state \
  ghcr.io/didi6135/claudify:latest
```

The `-v claudify-state:/state` binding gives the container a persistent volume that survives `docker rm` + `docker run`. State (persona, conversation log, OAuth) lives there.

### What runs the container in production?

For codaki.com (separate project, future):
- Backend service signs up Alice
- Backend allocates a unique container name + volume
- Backend collects Alice's BOT_TOKEN + OAuth via the website's UI
- Backend runs `podman run … claudify:latest`
- Container persists via the volume; restarts on host reboot via `--restart=unless-stopped`
- Backend tracks "alice → container codaki-alice" in its own database

That's a separate task — this 3.4.9 just produces the *image* that codaki.com (or any operator) can run.

## Comparing modes

| Concern | Solo (bash install) | Container (docker run) |
|---|---|---|
| Process supervisor | systemd user unit | Docker daemon (or podman) |
| State location | `~/.claudify-<name>/` | `/state` (volume) |
| OAuth token capture | Interactive `claude setup-token` during install | Pre-captured (env var) |
| Resource limits | systemd directives (`MemoryMax=`, `TasksMax=`) | Docker flags (`--memory`, `--pids-limit`) |
| Filesystem isolation | Same-host user (none meaningful for multi-instance — see ADR 0006) | Container = namespace (kernel-enforced) |
| Logging | journalctl | docker logs |
| Update | `update.sh` (curl latest + re-run) | `docker pull && docker stop && docker run` |
| Uninstall | `uninstall.sh` | `docker rm -f && docker volume rm` |
| Multi-tenant on one host | Weak (file perms only) | **Strong (each container is its own namespace)** |

## Notes / risks

- **Image size — ~400 MB.** Manageable but worth tracking. Multi-stage build + Alpine base could cut to ~150 MB if it ever becomes a problem.
- **Bot's OAuth token comes via env var.** That means it ends up in `/proc/<pid>/environ` inside the container. Container isolation prevents host-side processes from reading it; root inside the container could. Acceptable for a single-tenant container model. Codaki.com would issue per-container short-lived tokens that get rotated.
- **`tini` as PID 1.** Handles signal forwarding properly. Without it, claude won't get clean SIGTERM on `docker stop`, leading to slower shutdowns and possible zombie processes.
- **`USER claudify` (uid 1000) inside the container.** Means the container can't write to the host volume mount unless the host directory is owned by uid 1000. Solo users `docker run -v $PWD/state:/state` need to know this. Document in install-container.md.
- **State volume permissions.** `chmod 700` on `/state` at first boot, `600` on `credentials.env` and `channels/telegram/.env`. Same as solo install.
- **No systemd inside the container.** This means none of the Phase 3.6 systemd hardening directives apply. Container's native isolation (namespaces, cgroups) is the equivalent. Resource limits via Docker flags. Document the parallel.
- **Dual-mode complexity is real.** Every change to install logic now needs to consider: does this affect both modes? `lib/boot.sh` and `install.sh main()` will diverge over time. Worth a periodic audit.

## Relationship with other tasks

- **3.4.5 (multi-instance refactor):** completes first. Container mode uses `INSTANCE_NAME="container"` as the single instance — multi-instance is a solo-only concept (one container = one bot, scale by running more containers).
- **3.4.7 (migration):** doesn't apply to container — first-boot of a fresh volume IS a fresh install. No legacy paths to migrate.
- **3.5 (backup/restore):** the container's `/state` volume IS the backup unit. `docker run --rm -v claudify-state:/state ubuntu tar czf - /state > backup.tgz`. Inverse for restore. Trivial integration.
- **3.6 (security hardening):** Tier-1 systemd directives don't apply to container mode (container handles it natively). Tier-2 (BindPaths) is dead in solo Ubuntu 24.04 anyway, irrelevant for container. Tier-3 (syscall filter) — Docker has `--security-opt seccomp=…` for the same effect; worth a follow-up task to ship a default seccomp profile with the image.
- **Phase 4 (memory MCP, persona, manage MCP):** all live in `lib/` or `src/mcp/`, run identically in both modes. The manage MCP especially is what makes a hosted container useful — Alice talks to her bot to install skills.
