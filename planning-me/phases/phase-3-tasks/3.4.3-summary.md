## Design notes

| Decision | Why |
|---|---|
| **8-function contract, not 6** | `engine_seed_state` and `engine_install_channel_plugin` are real engine concerns â€” Claude Code needs both, future engines that don't can implement as no-ops. Cleaner than scattering "is this the kind of engine that needs X?" checks at every callsite. |
| **`lib/layout.sh` separate from `lib/engine.sh`** | Layout (`CLAUDIFY_ROOT` etc.) is engine-agnostic â€” `~/.claudify/` exists regardless of which CLI we're driving. Putting it in the engine adapter would muddy the boundary; making a separate file documents intent. NPM_PREFIX *is* engine-specific (it's npm-only), so it lives in the adapter. |
| **Dispatcher is a no-op in dist mode** | `lib/engine.sh` checks `declare -f engine_install` before sourcing the adapter. In dist, build.sh has already concatenated the adapter, so the function is defined and the source is skipped. In dev mode, the adapter file is sourced. Single dispatcher works in both modes. |
| **`engine_run_args` echoes the FULL ExecStart line** | Including the `script(1)` wrapping. Different engines might not need PTY wrapping (or might need a different wrapper); each adapter owns the full line. `service.sh` just embeds whatever `engine_run_args` returns. |
| **Engine-specific env-var names stay internal** | `CLAUDE_CODE_OAUTH_TOKEN` lives only in `claude-code.sh` (as `CLAUDE_OAUTH_ENV_VAR` constant). The orchestrator never sees it. A Gemini adapter would have its own internal env-var name; the abstraction is "engine_auth_setup persists creds to `$CREDS_FILE`", not "every engine uses an `ENGINE_TOKEN` variable". |
| **Step banner uses `$(engine_id)`** | Future-proofs status messages for non-Claude engines. "Authenticate claude-code" today; "Authenticate gemini" if/when that arrives â€” same code, different output. |

## Notes for next tasks

- **3.4.4 (manifest):** `engine_status` already returns `{engine, version, authenticated}` JSON. The manifest will embed a richer slice (instance metadata + channels + MCPs + skills + hooks); `engine_status` should become a sub-object inside the per-instance manifest.
- **3.4.5 (multi-instance):** `engine_run_args` will need to learn about per-instance channels (today: hardcoded `telegram`; tomorrow: read from `~/.claudify/instances/<name>/claudify.json`). The adapter signature stays the same; the function body grows to take the channel set as args or read it from the manifest.
- **Adding a 2nd engine:** drop `lib/engines/<id>.sh` implementing the 8 functions. Add an `--engine=<id>` install flag (or env var only â€” `CLAUDIFY_ENGINE=<id>` already works). No other code changes. ADR 0005's trigger conditions still gate when this is worth doing.
- **`engine_uninstall`:** today it's a no-op because the Claude binary is shared host-wide and per-bot state under `$CLAUDIFY_ROOT` is removed by `uninstall.sh`'s `rm -rf`. When 3.4.5 lands and instances live under `instances/<name>/`, this stays a no-op for shared binaries; engines that store per-instance state outside `$CLAUDIFY_ROOT` would put their cleanup here.
