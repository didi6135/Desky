## Goal

Two new commands serialize one or all instances into a tarball and
restore them onto a fresh server. **First production TypeScript** in
Claudify — sets the pattern for future TS modules.

After this task: server migration becomes a 2-command flow. Backup
runs nightly via cron in Phase 5; restore runs manually when needed.

## Why

`docs/architecture.md §10` lists backup/restore as Phase 3 closure.
ADR 0005 commits us to TypeScript for things that exceed bash's
comfort zone. Tarball-with-manifest-merge is exactly that case — JSON
manipulation, proper error types, file-permission preservation.

## Behavior — `backup.sh`

```bash
backup.sh --name <NAME>                # one instance
backup.sh --all                        # all instances (one tarball each)
backup.sh --name <NAME> --out <DIR>    # custom output dir (default: cwd)
```

Tarball name: `claudify-<instance>-<host>-<YYYYMMDD-HHMMSS>.tar.gz`

Tarball contents:
```
.claudify/
├── instances.json                   (the relevant entry only)
└── instances/<name>/
    └── (full instance state)
.config/systemd/user/
└── claudify-<name>.service
.claude/
└── claude.json.trust-slice         (just the projects[<workspace-path>] entry)
```

Permissions preserved.

## Behavior — `restore.sh`

```bash
restore.sh <tarball>                   # restore to defaults
restore.sh <tarball> --name <NEW>      # restore under a different name
```

Steps:
1. Extract to a temp dir
2. Read the included `claudify.json` to get the original name
3. If `--name <NEW>` provided, rewrite the manifest's name field + adjust paths
4. Refuse if `~/.claudify/instances/<name>/` already exists (would overwrite)
5. Move state into place; preserve perms
6. Merge `.claude/claude.json.trust-slice` into the existing `~/.claude.json`
7. Install the systemd unit; `daemon-reload`; `enable`; `start`
8. Run doctor for that instance; print result

## Steps

1. **`src/lib/log.ts`** — port bash `ui.sh`'s color helpers to TS. Step / ok / warn / fail. Banner. `setupLogging(filePath)` for file logging.
2. **`src/lib/state.ts`** — typed reads/writes for `~/.claudify/instances/<name>/`. `Instance` type. `findInstance(name)`. `listInstances()`.
3. **`src/lib/manifest.ts`** — typed wrapper around `instances.json` and per-instance `claudify.json`. Mirrors `lib/manifest.sh` API; one-to-one bash-to-TS mapping for testability.
4. **`src/lib/systemd.ts`** — `systemctlUser(verb, unit)`, `daemonReload()`, `serviceExists(name)`. Each just spawns the bash command + parses output.
5. **`src/backup.ts`** — entrypoint. Argv parsing (use `bun`'s built-in or a tiny dep). Orchestrate: collect, tar, write file. Print summary.

   set -euo pipefail
   if ! command -v bun >/dev/null 2>&1; then

   fi
   exec bun run "$(dirname "$(readlink -f "$0")")/src/$(basename "$0" .sh).ts" "$@"

## Notes / risks

- TS-vs-bash boundary: bash shims must work even if Bun is missing (clear error message + how to install). They're not allowed to call into TS without first checking.
- File permissions: tar's default behavior preserves perms when run as the same user. Test this — restore mustn't leak chmod 600 secrets to chmod 644.
- Trust-slice merge for `~/.claude.json` is the trickiest part: we're modifying a file Claude Code owns. Use `manifest_atomic_write` discipline. Test the failure case (corrupt input file).
- `bun build --target=node` may be needed to bundle for distribution. Investigate; if straightforward, use it. If not, ship the TS sources and let bun JIT them on the target (which already has bun installed).
