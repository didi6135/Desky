# Task 3.4.5.1 — Skill data dir + `lib/memory.sh` — Summary

**Status:** ✅ DONE 2026-05-12
**Spec:** [3.4.5.1-skill-data-dir.md](3.4.5.1-skill-data-dir.md)
**Effort:** ~45 min (as estimated)
**Unblocks:** Phase 4.0a (memory MCP — knows where to write its files), Phase 4.x skills (every skill that ships gets a known data path)

## What landed

The **storage substrate for skills** — ahead of any actual skill. Two
moving parts: a per-skill data directory at
`~/.claudify-<name>/data/<skill-id>/` (mode 700, persists across
`update.sh` and `--reset-config`) and a `lib/memory.sh` module that
resolves the path, exports it as `${CLAUDIFY_SKILL_DATA}`, and
manifest-checks declared write/read permissions. Future skills can
write `${CLAUDIFY_SKILL_DATA}/foo.db` and trust the path exists, is
private to their skill, and survives upgrades — matching Anthropic's
`${CLAUDE_PLUGIN_DATA}` convention so cross-ecosystem skills work.

## The contract — 5 functions

| Function | Args | Behaviour |
|---|---|---|
| `memory_dir` | `<skill-id>` | Echo `$CLAUDIFY_INSTANCE_DIR/data/<id>`; `mkdir -p` + `chmod 700` it. Idempotent. |
| `memory_path` | `<skill-id> <filename>` | Echo `<memory_dir>/<filename>` — convenience for SQLite-style usage. |
| `memory_assert_write` | `<skill-id> <db-name>` | Reads the manifest's `skills[].memory.writes` for the skill; non-zero + stderr message if `<db-name>` isn't listed. Accepts string OR array form in the manifest. |
| `memory_assert_read` | `<skill-id> <db-name>` | Same, against `memory.reads`. |
| `memory_export_env` | `<skill-id>` | `export CLAUDIFY_SKILL_DATA="$(memory_dir <skill-id>)"`. The agent calls this just before invoking a skill. |

Asserts are **accident prevention** (typo guard for db names), not a
sandbox — single-user trust model per ADR 0006. Failure mode is
"refused with a useful message", not "raised an alarm".

## Manifest schema gain

Per-instance `claudify.json` now carries skills as keyed-by-id entries
with optional memory declarations:

```json
"skills": [
  {
    "id": "reminders",
    "installed_at": "2026-05-12T06:18:39Z",
    "memory": {
      "writes": "reminders.db",
      "reads": ["persona.db"]
    }
  }
]
```

Both `writes` and `reads` accept either a string (one db) or an array
(multiple). Skills without a `memory` block are valid — they just
can't pass the assert helpers.

## Design choices

| Decision | Why |
|---|---|
| **`mkdir -p` + `chmod 700` inside `memory_dir`, not at install-time** | The instance-level `data/` is created at install (mode 755 — operator can navigate). Per-skill subdirs are created lazily when a skill first asks. Privacy applies at the per-skill boundary, not the parent. |
| **Asserts use stderr + return 1, not `fail`** | `fail` (from `lib/ui.sh`) calls `exit 1`. Bats tests need to source `memory.sh` in isolation — so the asserts can't depend on ui.sh. Plain stderr + non-zero return is just as actionable and stays bats-testable. |
| **`writes` / `reads` accept string OR array** | The spec showed `"writes": "reminders.db"` (single string), but the assert helper says "include" (collection-shaped). Normalizing both forms in jq (`if type == "array" then .[] else . end`) supports the natural single-string case without forcing every manifest to be array-shaped. |
| **`CLAUDIFY_SKILL_DATA` matches Anthropic's `CLAUDE_PLUGIN_DATA`** | Same shape, different name. Skills authored against Claude Code's plugin model can use either env var with no code change beyond a fallback chain. |
| **No daemon / broker / IPC** | File-level isolation (different dirs, different SQLite files, different mode bits) is enough for the single-user trust model per ADR 0006. The asserts are typo guards, not adversary defence — so we don't need a broker process to enforce them. |
| **`manifest_set_skill` is idempotent on `id`** | Re-running install (or a future `update.sh`) shouldn't duplicate skill entries. The jq expression looks up the existing index, replaces in-place if found, appends only if new. |

## Notes for next tasks

- **Phase 4.0a (memory MCP):** the MCP server writes `persona.db` — exact path is `$(memory_path persona persona.db)` once the persona skill is registered with `manifest_set_skill ... '"persona.db"'`. The MCP server itself is an engine-agnostic Bun/TypeScript binary; it accepts the data dir as an env var (`CLAUDIFY_SKILL_DATA`).
- **Phase 4.0b (engine wiring):** the real `engine_memory_setup` body for Claude Code will register the MCP via `claude mcp add claudify-memory <path>`. The path lives at `$CLAUDIFY_INSTANCE_DIR/bin/claudify-memory.js` — not under `data/` (data/ is for runtime state, not binaries).
- **Cross-skill reads:** the assert helper checks "is `<db-name>` listed in `memory.reads`?" but doesn't enforce *which* skill owns that DB. Convention: db-names are unique across the system (e.g. only the persona skill owns `persona.db`). If we ever want to disambiguate `reminders/notes.db` vs `journal/notes.db`, the schema would need to grow `reads: [{from: "persona", db: "persona.db"}]` — out of scope today.
- **3.4.6 (personal cmd):** when shipping `<instance> install-skill <id>`, that handler calls `manifest_set_skill` to register and `memory_dir` to verify the dir exists. Both helpers are ready.
