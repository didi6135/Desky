# Task 3.4.7 — Migration logic (single → multi-instance)

**Status:** ⏳ pending
**Estimated effort:** ~30 min
**Depends on:** 3.4.5 (multi-instance layout exists), 3.4.6 (personal cmd exists)
**Blocks:** 3.4.8 (docs)

## Goal

Operators with the *current* Claudify install (single-instance, state
at `~/.claudify/...` directly) seamlessly upgrade to the multi-instance
layout (state at `~/.claudify/instances/default/...`) when they next
run `update.sh`. Their bot keeps working; their tokens, allowlists,
and persona file are preserved.

## Why

Without migration, every existing user's update breaks until they
manually wipe and re-install. That's unacceptable. `architecture.md
§3f` calls this out as a Phase 3.4 responsibility.

## Scope

### Files to modify

- `install.sh` — runs the migration check + migration before any other install action
- `lib/manifest.sh` — adds `migrate_single_to_multi` helper

### Files to create

- `tests/bash/migration.bats` — assertion: given a fake old layout, after migration the new layout is correct and the old layout is gone

### Files to delete

- none

## What "old layout" looks like (what we migrate FROM)

```
~/.claudify/
├── credentials.env              # CLAUDE_CODE_OAUTH_TOKEN
├── workspace/
│   └── CLAUDE.md
├── telegram/
│   ├── .env
│   └── access.json
└── (no instances.json, no instances/ dir)

~/.config/systemd/user/
└── claude-telegram.service      # OLD service name
```

## What "new layout" looks like (what we migrate TO)

```
~/.claudify/
├── instances.json               # registry, with default entry
└── instances/
    └── default/
        ├── claudify.json
        ├── credentials.env
        ├── workspace/
        │   └── CLAUDE.md
        ├── channels/telegram/   # note: channels/ subdir is also new
        │   ├── .env
        │   └── access.json
        └── ...

~/.config/systemd/user/
└── claudify-default.service     # NEW service name (unit content also rewritten)

~/.local/bin/
└── default                      # personal command wrapper (3.4.6 generated this)
```

## Detection

`migrate_single_to_multi`:
1. Returns 0 (no-op) if `~/.claudify/instances.json` exists
2. Returns 0 (no-op) if `~/.claudify/` doesn't exist (first install)
3. Otherwise: detected old layout. Run migration.

Detection conditions for "old layout":
- `~/.claudify/credentials.env` exists, AND
- `~/.claudify/instances/` does NOT exist, AND
- (`~/.claudify/telegram/` OR `~/.claudify/workspace/`) exists

If detection is ambiguous (e.g., partial state from a half-broken
install), abort with a clear message — don't try to be clever.

## Migration steps

```
echo "Detected old single-instance layout. Migrating to ~/.claudify/instances/default/..."

# 1. Stop the old service
systemctl --user stop claude-telegram 2>/dev/null
systemctl --user disable claude-telegram 2>/dev/null

# 2. Move state into new location
mkdir -p ~/.claudify/instances/default
mv ~/.claudify/credentials.env  ~/.claudify/instances/default/credentials.env
mv ~/.claudify/workspace        ~/.claudify/instances/default/workspace
mkdir ~/.claudify/instances/default/channels
mv ~/.claudify/telegram         ~/.claudify/instances/default/channels/telegram
# (other dirs the old layout might have — best-effort with `mv` and 2>/dev/null guard)

# 3. Remove the old systemd unit
rm -f ~/.config/systemd/user/claude-telegram.service
systemctl --user daemon-reload

# 4. Hand control back to install.sh's normal path:
#    - It will write the new ~/.claudify/instances/default/claudify.json manifest
#    - Generate the new claudify-default.service unit
#    - Generate the personal command "default"
#    - Start the service

echo "Migration complete. Continuing with install..."
```

The crucial property: **migration is idempotent**. If it runs once and
the user re-runs `update.sh`, the second time the detection returns
no-op.

## Steps

1. Add `migrate_single_to_multi` to `lib/manifest.sh`:
   - Detection logic above
   - Move operations with explicit `mv -n` (don't overwrite if target exists — bug indicator)
   - Stop + disable + remove old systemd unit
   - `systemctl --user daemon-reload`
   - Print one summary line per moved item
2. `install.sh` main() — call `migrate_single_to_multi` as the first step after `parse_args`, before any other write. Critical: migration runs BEFORE `manifest_init_registry` and the rest of install.
3. CHANGELOG entry (under `### Changed`):
   ```
   Existing single-instance installs auto-migrate to the new multi-instance
   layout on next update.sh run. Old service `claude-telegram.service` is
   stopped and replaced by `claudify-default.service`. State is preserved.
   ```
4. Add `tests/bash/migration.bats`:
   - Set up a fake old layout under a temp `$HOME`
   - Run `migrate_single_to_multi`
   - Assert: new paths exist with right content, old paths gone, no `claude-telegram.service` reference left
   - Run again; assert it's a no-op
5. **Station11 round-trip — the real test:**
   - Get Station11 into the OLD layout. Two ways:
     - (a) Check out a Claudify commit from before 3.4.5, run install. Then check out main, run update.
     - (b) Manually re-create the old layout from a current install (move files back, rename service)
   - Run `bash ~/test/update.sh`
   - Assert: bot still replies; doctor 28/28 green; new paths in place; old paths gone
6. CHANGELOG: see above

## Acceptance criteria

- [ ] `migrate_single_to_multi` detects old layout reliably (no false positives on fresh installs or already-migrated state)
- [ ] Migration moves `credentials.env`, `workspace/`, and `telegram/` into the new locations
- [ ] Old `claude-telegram.service` is stopped, disabled, removed; `daemon-reload` runs
- [ ] After migration + normal install path, `claudify-default.service` is the active unit
- [ ] Bot still replies on Station11 after migration
- [ ] `default doctor` 28/28 green
- [ ] `tests/bash/migration.bats` covers happy path + idempotency
- [ ] CHANGELOG entry in Unreleased

## Test plan

- **Local bats:** simulate old layout under a temp HOME; verify migration shape
- **Station11:** stage old layout (manually move state back to pre-multi paths + rename service), then run `update.sh`. Verify the bot survives.

## Out of scope

- Migration in the OPPOSITE direction (multi → single). We won't support it; if anyone wants a single instance going forward they can `--name default` and ignore the registry.
- Automatic migration of old backup tarballs (3.5 will produce backups in the new format from day one)
- Multiple-instance migration scenarios (always single → default; only one old layout exists)

## Notes / risks

- This is the riskiest task in Phase 3.4 — operators have real state with real tokens. Test idempotency exhaustively. A bad migration corrupts the install.
- If the user has BOTH old layout AND new layout (e.g., partial reinstall), refuse to migrate and surface a "this looks ambiguous, please contact support / open an issue" message. Don't risk data loss.
- The migration runs early in `install.sh` — but printing must go through `setup_logging` so it lands in the log file. Order matters: `parse_args → setup_logging → migrate_single_to_multi → ...`.
- After migration, the personal command (`default`) is generated by the existing 3.4.6 path. If 3.4.6 is integrated correctly, no extra work here.
